# TC-E2E-024 — Deploy Strategy: Canary Release + Rollback for a Production Data Pipeline

## Summary

A new production data pipeline (`pipeline_order_enrichment_v2`) is ready for deployment. It replaces a legacy enrichment pipeline with a redesigned processing graph, a new deduplication strategy, and an updated schema mapping. Given the pipeline's downstream impact on revenue reporting and customer segmentation models, a standard full-cutover deploy is too risky. Instead, this response defines a **canary release** strategy: route a small percentage of traffic to the new pipeline first, monitor against defined SLO guardrails, and either promote to full traffic or trigger an automated rollback.

A complete runbook is included covering deploy steps, monitoring, and rollback procedure.

---

## Decision

Execute a canary **deploy** using a traffic-split approach: route **5% of incoming data** through `pipeline_order_enrichment_v2` while 95% continues through the legacy pipeline. Hold the canary for a minimum of 4 hours (or until 10,000 events have been processed through it). Promote to 100% only if all SLO guardrails remain green throughout the observation window.

The runbook for this release must be reviewed and acknowledged by the on-call engineer before the deploy begins.

---

## Rationale

### Why Canary Over Blue/Green or Direct Cutover

| Deploy Strategy | Risk | Recovery Time | Data Integrity Risk |
|---|---|---|---|
| Direct cutover | High | Minutes to hours | High — all traffic affected immediately |
| Blue/green (full parallel) | Medium | Seconds (swap) | Medium — requires dual infrastructure cost |
| Canary 5% → 100% (chosen) | Low | Seconds (reroute) | Low — only 5% of data affected during observation |
| Shadow mode (read-only) | Very low | N/A | None — but cannot validate write-path behavior |

A **canary** is preferred because it validates real production data behavior — not synthetic test data — while limiting blast radius to 5% of volume.

### Traffic Split Implementation

For a Kafka-based pipeline, the canary split is implemented via consumer group partition assignment:

```python
# Airflow DAG — canary routing via Kafka partition assignment
# Partitions 0–47 (95%) → legacy consumer group
# Partitions 48–49 (5%) → canary consumer group

CANARY_PARTITIONS = [48, 49]
LEGACY_PARTITIONS = list(range(0, 48))

canary_task = KafkaConsumerOperator(
    task_id="canary_enrichment_v2",
    topic="orders.raw",
    consumer_group="order_enrichment_v2_canary",
    partitions=CANARY_PARTITIONS,
    processing_fn=enrich_order_v2,
    output_topic="orders.enriched.canary",
    dag=dag,
)
```

For batch pipelines (Airflow + BigQuery), canary is implemented by processing a deterministic 5% sample:

```sql
-- Canary batch: process only rows where MOD(ABS(FARM_FINGERPRINT(order_id)), 100) < 5
INSERT INTO warehouse.fact_orders_canary
SELECT *
FROM staging.stg_orders_raw
WHERE MOD(ABS(FARM_FINGERPRINT(order_id)), 100) < 5
  AND batch_date = '{{ ds }}';
```

### SLO Guardrails

The following guardrails must be monitored throughout the canary window. A single P1 breach triggers immediate automated rollback:

| Guardrail | SLO Threshold | Window | Priority |
|---|---|---|---|
| Data freshness | Canary output lag ≤ legacy lag + 5 minutes | Rolling 15 min | P2 |
| Row count parity | Canary processed rows ≥ 98% of expected (based on partition %) | Per 5-min window | P2 |
| Error rate | Pipeline error rate ≤ 0.1% of records | Rolling 15 min | P1 |
| Schema validation failures | Zero schema validation errors | Per record | P1 |
| Null rate on key fields | Null rate on `account_id`, `order_total_usd` ≤ 0.5% | Rolling 30 min | P2 |
| Processing latency (p99) | p99 end-to-end latency ≤ 120% of legacy p99 | Rolling 15 min | P2 |
| Downstream model row count | `mart_orders` row count delta ≤ 0.1% vs previous run | Per dbt run | P2 |

### Automated Rollback Trigger Conditions

Rollback is triggered automatically if:
1. Any P1 guardrail is breached (error rate > 0.1%, any schema validation failure).
2. Two or more P2 guardrails breach simultaneously within a 5-minute window.
3. The on-call engineer explicitly triggers rollback via the runbook.

```python
from airflow.operators.python import BranchPythonOperator

def check_canary_guardrails(**context):
    metrics = fetch_canary_metrics(window_minutes=15)
    if metrics["error_rate"] > 0.001:
        return "rollback_canary"
    if metrics["freshness_lag_seconds"] > 300:
        return "rollback_canary"
    return "promote_canary"

guardrail_check = BranchPythonOperator(
    task_id="check_canary_guardrails",
    python_callable=check_canary_guardrails,
    dag=dag,
)
```

## Runbook

### Section 1: Pre-Deploy Checklist

- [ ] All unit tests pass in CI (`pytest tests/unit/`)
- [ ] Integration tests pass against staging environment
- [ ] Schema compatibility check passes in staging schema registry
- [ ] Canary output table `fact_orders_canary` exists and is empty
- [ ] Datadog monitors for canary guardrails are active
- [ ] On-call engineer has acknowledged the deploy in PagerDuty
- [ ] Rollback SQL script is ready and tested in staging
- [ ] Stakeholders notified via Slack `#data-platform-deploys`

### Section 2: Deploy Steps

```bash
# Step 1: Tag the release
git tag -a v2.0.0-canary -m "Canary deploy: order_enrichment_v2"
git push origin v2.0.0-canary

# Step 2: Deploy canary Airflow DAG (paused by default)
airflow dags trigger pipeline_order_enrichment_v2_canary \
  --conf '{"canary_pct": 5, "output_table": "fact_orders_canary"}'

# Step 3: Unpause canary DAG
airflow dags unpause pipeline_order_enrichment_v2_canary

# Step 4: Monitor guardrails for 4 hours minimum
# → Watch Datadog dashboard
# → Watch dbt Cloud run results for canary tests
# → Verify row counts in fact_orders_canary

# Step 5a (all guardrails green after 4h): Promote to 100%
airflow dags trigger pipeline_order_enrichment_v2_promote \
  --conf '{"promote_pct": 100}'

# Step 5b (any guardrail breach): Execute rollback — see Section 3
```

### Section 3: Rollback Procedure

**Rollback SLA**: All rollback steps must complete within 10 minutes of the trigger condition being detected.

```bash
# Step 1: Pause canary DAG immediately
airflow dags pause pipeline_order_enrichment_v2_canary

# Step 2: Delete canary output data to prevent contamination
bq query --use_legacy_sql=false \
  "DELETE FROM warehouse.fact_orders_canary WHERE batch_date = CURRENT_DATE()"

# Step 3: Confirm legacy pipeline is processing all traffic
airflow dags unpause pipeline_order_enrichment_v1  # if it was paused

# Step 4: Notify stakeholders
# Post to #data-platform-deploys: "Canary rollback executed at HH:MM UTC.
# Guardrail [NAME] breached. Legacy pipeline processing 100% of traffic.
# Post-mortem scheduled for [DATE]."
```

### Section 4: Release Gates for Full Promotion

Before promoting from canary to 100%, all of the following gates must pass:

- [ ] All SLO guardrails green for the full 4-hour observation window
- [ ] Canary row count matches expected volume within 1% tolerance
- [ ] No PagerDuty incidents opened during canary window
- [ ] Manual spot-check: 5 random `order_id` values verified identical in both legacy and canary output
- [ ] Data platform lead has approved promotion in the deploy JIRA ticket

## Trade-offs

| Decision | Pros | Cons |
|---|---|---|
| 5% canary split (chosen) | Low blast radius; real production validation | Slower to detect rare edge cases occurring in <5% of records |
| 10% canary split | Faster validation, catches more edge cases | 2× blast radius compared to 5% |
| 1% canary split | Minimal blast radius | Very slow to accumulate statistical significance; may miss volume-dependent issues |
| Shadow mode only | Zero production impact | Does not validate write path, deduplication, or downstream dependency behavior |
| 4-hour minimum observation window | Captures intra-day traffic pattern variation | Delays full promotion; may push against downstream SLA windows |

**SLO guardrail sensitivity**: The 0.1% error rate threshold is intentionally tight for a canary. In full production the SLO may be set at 0.5%. Tighter thresholds during canary reduce the chance of promoting a broken pipeline, at the cost of more false-positive rollbacks. Calibrate against historical baseline error rates before each major release.

## Next Steps

1. **Execute the pre-deploy checklist** (Section 1 of the runbook) at least 24 hours before the canary deploy. Resolve any blockers before proceeding.

2. **Schedule the canary deploy** during a low-traffic window (Tuesday–Thursday, 10:00–14:00 UTC) to maximize the chance of catching issues before peak hours.

3. **Set up Datadog canary dashboard** with all seven guardrail monitors wired to the canary output table and consumer group. Share the dashboard link in the deploy Slack notification.

4. **After successful promotion**: Archive the `fact_orders_canary` table for 7 days (audit trail), then drop it. Remove the canary DAG. Update the pipeline documentation to reflect v2 as the production pipeline.

5. **Post-deploy retrospective** (regardless of outcome): Document what went well, what guardrails fired (if any), and whether the observation window was correctly sized. Feed learnings back into the deploy runbook template.

6. **Automate this pattern**: If this canary deploy succeeds, codify the guardrail check logic and traffic-split mechanism as a reusable Airflow plugin for all future major pipeline releases.
