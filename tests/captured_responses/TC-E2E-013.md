## Summary

The submitted SQL contains two critical issues: a bare `INSERT` that is not idempotent and will produce duplicate rows on any re-run, and a correlated subquery inside a large-table scan that degrades to O(n²) complexity. Both are classified as anti-patterns per Principle 1 (Idempotency first) and must be fixed before this query is deployed to production.

## Decision

Replace the bare `INSERT` with a `DELETE + INSERT` or `MERGE` pattern scoped to the target partition. Rewrite the correlated subquery as a `JOIN` or window function. Run `EXPLAIN` on the revised query before deploying.

## Rationale

**Non-idempotent INSERT:** A bare `INSERT INTO fact_orders SELECT ... FROM staging_orders` appends rows every time the query runs. If the pipeline retries (e.g., after an Airflow task failure), the partition will contain duplicate rows. This violates Principle 1: Idempotency first. The correct fix is:

```sql
-- Option A: DELETE + INSERT (partition-scoped, idempotent)
DELETE FROM fact_orders WHERE order_date = '{{ ds }}';

INSERT INTO fact_orders (order_id, customer_id, total_amount, order_date)
SELECT order_id, customer_id, total_amount, order_date
FROM staging_orders
WHERE order_date = '{{ ds }}';

-- Option B: MERGE (handles upserts for dimension/fact tables with updates)
MERGE INTO fact_orders AS target
USING staging_orders AS src
ON target.order_id = src.order_id
WHEN MATCHED THEN UPDATE SET
    target.total_amount = src.total_amount,
    target.order_date   = src.order_date
WHEN NOT MATCHED THEN INSERT (order_id, customer_id, total_amount, order_date)
    VALUES (src.order_id, src.customer_id, src.total_amount, src.order_date);
```

**Correlated subquery:** The pattern `WHERE x IN (SELECT ... FROM large_table WHERE outer_col = ...)` re-executes the inner query for every row in the outer scan. Rewrite as a `JOIN` or window function:

```sql
-- Correlated subquery (anti-pattern):
SELECT o.*
FROM fact_orders o
WHERE o.customer_id IN (
    SELECT customer_id FROM staging_orders WHERE order_date = o.order_date
);

-- Fixed: JOIN (single pass, optimizer can hash-join):
SELECT o.*
FROM fact_orders o
JOIN (
    SELECT DISTINCT customer_id, order_date FROM staging_orders
) s ON o.customer_id = s.customer_id AND o.order_date = s.order_date;
```

## Trade-offs

| Option | Pros | Cons | When to Use |
|--------|------|------|-------------|
| DELETE + INSERT | Simple, explicit, easy to audit | Two statements (wrap in transaction) | Partition-scoped full refresh |
| MERGE | Single statement, handles updates | Slightly more complex syntax | Dimension/fact tables that receive updates |
| Bare INSERT (current) | Simplest to write | Not idempotent — duplicates on retry | Never in scheduled pipelines |
