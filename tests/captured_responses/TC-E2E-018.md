# TC-E2E-018 — SPARK Mode: Data Skew Mitigation on a Customer ID Join

**Mode:** SPARK
**Scenario:** Diagnosing and fixing data skew on a `customer_id` join in a PySpark job where one customer has 40% of all rows.
**Date Captured:** 2026-03-25

---

## Summary

A PySpark ETL job that joins a `transactions` DataFrame (~4 billion rows) with a `customer_profiles` DataFrame (~50 million rows) on `customer_id` is exhibiting severe tail latency in production. The job nominally completes in approximately 8 minutes when run on balanced test data but consistently takes 55–65 minutes in production, with the Spark UI showing the exchange (shuffle) stage stalled on a single task that never finishes within the expected window. The remaining 199 of 200 tasks in the shuffle stage complete within 3–4 minutes, but the job cannot advance until the final task completes, which processes roughly 1.6 billion rows — 40% of the entire dataset — due to a single enterprise customer (`customer_id = 'ACME_CORP'`) that dominates the transaction volume.

This is a textbook data skew problem. When Spark performs a sort-merge join on `customer_id`, it partitions both DataFrames by hash of `customer_id` and routes all rows for each distinct key to the same reducer task. Because `ACME_CORP` accounts for 40% of all rows, every row with that key lands in one partition — one task — while the other 199 tasks each handle a roughly equal share of the remaining 60%. The skewed task must process 1.6 billion rows solo while all other tasks are long done. The job's total wall-clock time is essentially determined by the single most skewed task.

Spark's Adaptive Query Execution (AQE) framework, introduced in Spark 3.0, includes an automatic skew join optimization (`spark.sql.adaptive.skewJoin.enabled = true`) that can detect and split skewed partitions at runtime. However, the current cluster configuration has AQE disabled, and even with AQE enabled, the automatic threshold may not correctly detect a single-key skew of this magnitude without tuning. Two additional mitigation techniques are applicable: (1) salting — artificially distributing the hot key across multiple partitions by appending a random integer suffix to `customer_id` on both sides of the join — and (2) a broadcast join for the `customer_profiles` dimension table, which eliminates the shuffle entirely by sending the smaller table to every executor.

The preferred solution is a broadcast join for `customer_profiles` (50 million rows, approximately 2 GB serialized), which avoids the shuffle stage entirely and makes skew irrelevant. If the dimension table is too large to broadcast reliably, salting with AQE enabled is the recommended fallback. Both approaches are described in full below with code examples.

---

## Decision

Apply the following skew mitigations in priority order:

1. **Primary fix: Broadcast join for `customer_profiles`.** Force a broadcast hint on the `customer_profiles` DataFrame. At ~2 GB serialized, it fits comfortably within the default broadcast threshold (10 GB) and within executor memory on the current cluster (each executor has 16 GB heap). A broadcast join sends a copy of `customer_profiles` to every executor, eliminating the shuffle stage and making `customer_id` key distribution irrelevant. This is the simplest and most effective fix.

2. **Secondary fix (if broadcast is not viable): Salting + AQE.** If `customer_profiles` grows beyond broadcast limits in the future, implement a salting strategy that adds a random salt integer (0–9) to `ACME_CORP`'s `customer_id` on both sides of the join, distributing its rows across 10 partitions. Enable AQE with skew join detection as an additional layer of adaptive optimization.

3. **Always enable AQE.** Regardless of which join strategy is used, enable `spark.sql.adaptive.enabled = true` and `spark.sql.adaptive.skewJoin.enabled = true` on the cluster. This provides automatic skew detection and mitigation for all joins in the job, not just the `customer_id` join, and reduces the need for manual per-join tuning.

## Rationale

**Detecting skew via Spark UI and partition size checks.** Before implementing any fix, confirm the skew diagnosis with data. In the Spark UI, navigate to the Stages tab for the exchange stage and sort tasks by "Duration" descending. A skewed job will show one or a few tasks with dramatically higher duration and shuffle read bytes than the median task. In this case, Task #137 shows 1.62 TB shuffle read and 58 minutes duration, while the median task shows 8.2 GB shuffle read and 2.8 minutes duration — a 197x size imbalance.

Programmatically, the skew can be confirmed with a partition size check before the join:

```python
from pyspark.sql import functions as F

# Check key distribution
skew_check = (
    transactions
    .groupBy("customer_id")
    .count()
    .orderBy(F.desc("count"))
)
skew_check.show(10)
# Output shows ACME_CORP: 1,621,000,000 rows (40.5% of 4B total)
```

**Broadcast join implementation.** The broadcast join is the cleanest solution when the smaller table fits in memory. PySpark's `broadcast()` hint tells the Spark planner to use a BroadcastHashJoin strategy, which serializes the smaller DataFrame and sends it to every executor node's memory. Each executor can then perform a local hash lookup against `customer_id` without any shuffle. This completely bypasses the hash-partition-and-shuffle stage that creates the skew bottleneck.

```python
from pyspark.sql.functions import broadcast

result = transactions.join(
    broadcast(customer_profiles),
    on="customer_id",
    how="left"
)
```

The broadcast hint is a suggestion, not a command — if the DataFrame exceeds `spark.sql.autoBroadcastJoinThreshold` (default 10 MB, should be raised to `10g` for this use case), Spark will ignore the hint and fall back to a sort-merge join. Set the threshold explicitly:

```python
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "10g")
```

**Salting technique for unbounded hot keys.** If `customer_profiles` cannot be broadcast (e.g., it grows to 50+ GB), salting is the next best option. The idea is to replicate each row in `customer_profiles` S times (S = salt factor, typically 10–20), each with a different salt value appended to `customer_id`. On the transactions side, add a random salt integer (0 to S-1) to each row's `customer_id`. The join then distributes `ACME_CORP`'s rows across S partitions rather than 1, reducing the maximum partition size by a factor of S.

```python
import pyspark.sql.functions as F

SALT_FACTOR = 10

# Salt the hot key on the large table
transactions_salted = transactions.withColumn(
    "customer_id_salted",
    F.when(
        F.col("customer_id") == "ACME_CORP",
        F.concat(F.col("customer_id"), F.lit("_"), (F.rand() * SALT_FACTOR).cast("int").cast("string"))
    ).otherwise(F.col("customer_id"))
)

# Explode the dimension table to match each salt value
from pyspark.sql.functions import array, explode, lit

salt_range = array([lit(i) for i in range(SALT_FACTOR)])
customer_profiles_salted = (
    customer_profiles
    .withColumn("salt", explode(salt_range))
    .withColumn(
        "customer_id_salted",
        F.when(
            F.col("customer_id") == "ACME_CORP",
            F.concat(F.col("customer_id"), F.lit("_"), F.col("salt").cast("string"))
        ).otherwise(F.col("customer_id"))
    )
    .drop("salt")
)

result = transactions_salted.join(
    customer_profiles_salted,
    on="customer_id_salted",
    how="left"
).drop("customer_id_salted")
```

Salting reduces the maximum partition size for `ACME_CORP` from 1.62 TB to approximately 162 GB (1/10th), which is manageable for the current executor configuration.

**AQE configuration for adaptive skew join handling.** Spark 3.x AQE can automatically detect skewed partitions and split them at runtime without requiring explicit salting. Enable it with:

```python
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")
# Optional: tune the skew detection threshold
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionFactor", "5")
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes", "256m")
```

AQE works by collecting runtime statistics after the shuffle and re-optimizing the plan before executing the join. If a partition exceeds `skewedPartitionFactor * median_partition_size` or `skewedPartitionThresholdInBytes`, AQE splits it into multiple sub-tasks and adjusts the join plan accordingly. For the extreme skew in this job (197x imbalance), AQE alone may not fully resolve the problem — the automatic split threshold defaults may not fire aggressively enough. Tuning `skewedPartitionFactor` down to `2` and `skewedPartitionThresholdInBytes` to `64m` makes AQE more aggressive for this use case.

## Trade-offs

| Approach | Skew Eliminated? | Shuffle Eliminated? | Implementation Effort | Memory Overhead | Scalability |
|---|---|---|---|---|---|
| No change (sort-merge join, no AQE) | No | No | None | Low | Poor — degrades as ACME_CORP grows |
| Enable AQE only (adaptive skew join) | Partially | No | Very Low | Low | Good for moderate skew |
| Broadcast join (recommended primary) | Yes | Yes | Low | Medium (2 GB per executor) | Good up to ~10 GB dimension table |
| Salting + AQE (recommended fallback) | Yes | No | Medium | Low | Excellent — scales with salt factor |
| Broadcast + salting (belt and suspenders) | Yes | Yes | Medium | Medium | Overkill for current scale |
| Repartition by composite key | Partial | No | Low | Low | Limited — doesn't fix single-key skew |

**Broadcast join limitations.** Broadcasting works well when the dimension table is small enough to fit in executor memory and network transfer time is acceptable. At 2 GB serialized, `customer_profiles` adds approximately 2 GB per executor for the duration of the job. With 40 executors on the current cluster, the total cluster memory overhead is 80 GB — acceptable given the 640 GB total cluster memory. However, if `customer_profiles` grows to 20+ GB, broadcast becomes expensive and eventually hits memory limits on executors. At that point, salting becomes the primary strategy.

**Salting complexity and correctness.** Salting is a manual intervention that works but adds significant code complexity. The salt must be applied consistently and correctly to both sides of the join — any mismatch results in dropped rows. Additionally, the salt column must be dropped after the join to avoid polluting the output schema. Salting also inflates `customer_profiles` by a factor of S in memory (each salt value creates a replicated row), which may be a concern if the dimension table is large. AQE-based automatic skew handling is preferred over manual salting where the skew factor is moderate (under 50x), but for 197x imbalance, manual salting with a salt factor of 10–20 is more reliable than relying on AQE's heuristics alone.

**AQE as a safety net, not a primary strategy.** AQE's automatic skew join optimization is valuable as a general-purpose safety net but should not be relied on as the primary fix for known, extreme, single-key skew. AQE applies heuristics based on partition size statistics collected at shuffle boundaries; for extremely skewed distributions, the heuristics may split the hot partition into too few sub-tasks or fail to detect the skew if the overall partition count is low. For this specific job, AQE should be enabled as a complement to the broadcast join, providing automatic protection for any other skewed joins in the pipeline that have not yet been identified.
