# TC-E2E-017 — SQL Mode: Incremental Late-Arriving Record Handling

**Mode:** SQL
**Scenario:** Designing an idempotent incremental SQL load pattern for an events table that receives late-arriving records up to 3 days after the event date.
**Date Captured:** 2026-03-25

---

## Summary

The `events` table receives a continuous stream of records from mobile clients, third-party webhooks, and internal microservices. A significant portion of these records arrive late — audit logging shows that roughly 12% of all events are delivered between 24 and 72 hours after their `event_timestamp`. The current incremental load job uses a simple `INSERT WHERE event_date = CURRENT_DATE - 1` pattern, which permanently misses any record whose `event_date` is correct but whose delivery timestamp falls outside yesterday's processing window. Over a 90-day period this has caused a 1.3% row-count shortfall in the `events` fact table compared to source system counts, leading to incorrect downstream KPI calculations for retention and funnel analysis.

The root cause is that the load job anchors its processing window on `event_date` (the business timestamp, server-side derived) rather than on `updated_at` (the row modification timestamp managed by the ingestion layer). Late-arriving records have correct `event_date` values — sometimes days in the past — but their `updated_at` values reflect when the ingestion layer finally processed them. By switching the watermark column from `event_date` to `updated_at`, the incremental job captures all records that were written or updated in the ingestion layer within the last N hours, regardless of how far in the past their `event_date` falls.

The watermark-based strategy requires two complementary mechanisms: (1) a correction window that reprocesses any record whose `updated_at` falls within the last 3 days (72 hours), and (2) an idempotent MERGE/upsert pattern that safely handles re-processing of records that were already loaded in a previous run. Without idempotent merge semantics, re-running the job over a 3-day correction window would produce duplicate rows for on-time records that were already loaded correctly. With a proper `MERGE` statement keyed on `event_id`, duplicate loads become no-ops for unchanged rows and updates for corrected rows.

Dedup logic is required as an additional safeguard because the upstream ingestion layer occasionally delivers the same `event_id` more than once (a known at-least-once delivery guarantee in the webhook infrastructure). A dedup CTE that selects `ROW_NUMBER() OVER (PARTITION BY event_id ORDER BY updated_at DESC) = 1` before the MERGE ensures that only the most recent version of each event_id enters the merge operation, preventing spurious updates from stale duplicate deliveries. Together, watermark strategy, idempotent MERGE, and dedup logic form a complete pattern for reliable incremental loading of late-arriving event data.

---

## Decision

Implement a watermark-based incremental load with a 3-day correction window, an idempotent MERGE upsert keyed on `event_id`, and a pre-merge dedup CTE. The job will:

1. **Use `updated_at` as the watermark column** — Select all source records where `updated_at >= CURRENT_TIMESTAMP - INTERVAL '3 days'`. This captures on-time records, late-arriving records (up to 72 hours late), and any corrections or re-deliveries within the window.

2. **Dedup before merging** — Apply a `ROW_NUMBER() OVER (PARTITION BY event_id ORDER BY updated_at DESC)` in a CTE and filter to `rn = 1`, keeping only the most recently updated version of each `event_id` in the candidate set.

3. **MERGE into target using `event_id` as the match key** — When a match is found, update all mutable fields (`event_properties`, `user_id`, `updated_at`) if the source `updated_at` is newer than the target. When no match is found, insert the new record. This pattern is fully idempotent: re-running the same job window twice produces identical target state.

4. **Track the high-water mark** — After each successful run, write the maximum `updated_at` value processed to a `pipeline_state` control table. On the next run, use this value as the lookback lower bound (subject to a minimum of `CURRENT_TIMESTAMP - INTERVAL '3 days'`) to avoid reprocessing unbounded history on restarts.

This approach handles late-arriving records up to 3 days old, supports backfill by running the job with an explicit date range override, and is safe to retry or replay without producing duplicates.

---

## Rationale

**Watermark column selection: `updated_at` over `event_date`.** The watermark is the column used to identify which source records are "new" since the last processing run. `event_date` reflects business logic (when the user action occurred) and is set by the client or application server. `updated_at` reflects ingestion infrastructure time (when the ingestion layer wrote or last modified the row) and is managed server-side with a reliable monotonically increasing value. Late-arriving records have old `event_date` values but fresh `updated_at` values. Using `updated_at` as the watermark column ensures the job sees every record the ingestion layer has touched within the correction window, regardless of business-time event date. This is the standard pattern for incremental loads on event tables where clients are unreliable or operate in environments with intermittent connectivity.

**Correction window sizing.** The 3-day (72-hour) correction window is chosen based on observed late-arrival distribution from the audit log. 98.7% of late-arriving records arrive within 72 hours of their `event_timestamp`. Setting the window to 72 hours captures nearly all late arrivals with a comfortable margin. A longer window (e.g., 7 days) would be safer against extreme outliers but increases the volume of data reprocessed on every run and extends merge operation runtime. The window size should be reviewed quarterly as client behavior changes. It should be stored as a configurable parameter (`@correction_window_days`) rather than a hard-coded literal to allow adjustment without code changes.

**Idempotent MERGE semantics.** The MERGE statement (also called `UPSERT` in some SQL dialects) is the cornerstone of idempotent incremental loading. It combines an INSERT for new rows and an UPDATE for existing rows in a single atomic operation, keyed on a business identifier (`event_id`). The critical detail for correctness is the update condition: the MERGE should only update a target row if the source `updated_at` is strictly greater than the current target `updated_at`. This prevents stale re-deliveries from overwriting newer data. A naive MERGE without this condition would overwrite correct, up-to-date target rows with older source versions during backfill or replay operations.

Example MERGE structure (BigQuery syntax):

```sql
MERGE events_target AS t
USING (
  SELECT *
  FROM (
    SELECT
      *,
      ROW_NUMBER() OVER (PARTITION BY event_id ORDER BY updated_at DESC) AS rn
    FROM events_source
    WHERE updated_at >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 3 DAY)
  )
  WHERE rn = 1
) AS s
ON t.event_id = s.event_id
WHEN MATCHED AND s.updated_at > t.updated_at THEN
  UPDATE SET
    t.user_id = s.user_id,
    t.event_properties = s.event_properties,
    t.updated_at = s.updated_at
WHEN NOT MATCHED THEN
  INSERT (event_id, event_date, user_id, event_properties, updated_at)
  VALUES (s.event_id, s.event_date, s.user_id, s.event_properties, s.updated_at)
```

**Dedup logic for at-least-once sources.** The upstream webhook infrastructure guarantees at-least-once delivery, meaning the same `event_id` may appear multiple times in the source with identical or slightly different field values (e.g., if a delivery retry fires after a network partition). Without dedup, the MERGE's candidate set may contain two rows with the same `event_id`, causing a `Multiple source rows match a single target row` error in strict MERGE implementations (BigQuery raises this as a runtime error). The `ROW_NUMBER()` dedup CTE resolves this by selecting only the highest `updated_at` version of each `event_id` before it reaches the MERGE, ensuring the candidate set has at most one row per `event_id`.

**Backfill support.** The same job supports full historical backfill by overriding the `@correction_window_days` parameter with a larger value (or explicit `@start_date`/`@end_date` bounds) and running the job manually. Because the MERGE is idempotent, running a backfill over a range that was already processed simply re-applies the same updates, resulting in no change for records that have not been modified since the last run. This makes backfill operations safe to run at any time without coordinating with downstream consumers.

## Trade-offs

| Approach | Late Arrival Coverage | Idempotent | Dedup Safe | Complexity | Cost per Run |
|---|---|---|---|---|---|
| Append-only (INSERT, no dedup) | None — misses late records entirely | No | No | Very Low | Low |
| Date-partition reload (DELETE + INSERT) | Full window, not targeted | Yes (if window is clean) | No | Low | Medium |
| Watermark + INSERT only (no MERGE) | Yes (72h window) | No — duplicates on retry | No | Low | Low |
| Watermark + MERGE (no dedup) | Yes (72h window) | Yes | No — MERGE error on dup source | Medium | Medium |
| Watermark + dedup CTE + MERGE (recommended) | Yes (72h window) | Yes | Yes | Medium | Medium |
| Full table reload nightly | Complete | Yes | Yes | Very Low | Very High |

**Watermark + MERGE vs. delete-and-reload.** A partition delete-and-reload pattern (delete all rows in the correction window partitions, then insert fresh from source) is simpler to implement and also idempotent, but it has two drawbacks: (1) it requires dropping and recreating partitions, which locks those partitions and can cause brief query unavailability for downstream consumers; (2) it cannot handle the case where late-arriving records fall in partitions that also contain records that were never late — the delete-and-reload would unnecessarily re-process all rows in those partitions. The MERGE pattern is more surgical: it only touches rows that exist in the source correction window, leaving all other rows untouched.

**3-day window vs. longer correction windows.** Increasing the correction window from 3 days to 7 days would capture a slightly larger tail of late arrivals (an additional ~0.3% based on audit data) but would increase the MERGE candidate set size by roughly 2.3x, extending job runtime proportionally. For most event pipelines, the additional coverage is not worth the runtime cost. A better alternative for extreme outliers (records arriving more than 3 days late) is a separate low-frequency backfill job that runs weekly with a 30-day window and is triggered manually or on a schedule outside peak hours.

**Append-only vs. MERGE for audit log use cases.** Some event tables serve as immutable audit logs where records must never be updated — only new records appended. In this pattern, late arrivals are simply appended as new rows (potentially with a `delivery_timestamp` column distinct from `event_timestamp`), and downstream queries are written to handle duplicates at query time. This is appropriate when the consumer requires full delivery history (including retries), but it is not recommended for fact tables used in KPI calculations because dedup logic must then be embedded in every downstream query rather than handled once at the load layer.

## Next Steps

1. **Implement and deploy the watermark + dedup + MERGE job** — Convert the existing append-only incremental job to the watermark-based MERGE pattern described above. Deploy to the staging environment first, run over the last 90 days of data, and compare row counts and field values against the source system. Target deployment to production: 2 weeks. Owner: pipeline team.

2. **Parameterize the correction window** — Store `correction_window_days = 3` in the pipeline configuration table and pass it as a query parameter. Document the procedure for temporarily extending the window during incident response or large-scale backfill operations.

3. **Backfill historical gap** — Run a one-time backfill using a 90-day correction window to recover the 1.3% row-count shortfall identified in the audit. Schedule during off-peak hours to minimize impact on concurrent query workloads. Validate row counts match source system after backfill completes.

4. **Add late-arrival monitoring** — Instrument the job to log the count of late-arriving records (rows where `event_date < CURRENT_DATE - 1` but `updated_at >= CURRENT_TIMESTAMP - INTERVAL 3 DAY`) per run. Alert if this count exceeds 5% of total rows processed in a single run, as a spike may indicate a source system delivery issue requiring investigation.

5. **Review and update downstream KPI queries** — Audit all downstream models that read from `events` to ensure they do not filter on `event_date = CURRENT_DATE - 1` (which would still miss late arrivals even after the pipeline fix). Any downstream query that uses a date filter must allow for a correction window or be updated to read from the finalized (post-correction) partition.

6. **Establish SLA for correction window closure** — Agree with data consumers on the SLA for when a given event date's data is considered "final." With a 3-day correction window, the recommended SLA is: data for `event_date = D` is finalized at `D + 4` (allowing 3 days for late arrivals plus 1 day for the correction job to process them). Document this in the data contract for the `events` table.
