# TC-E2E-022 — Schema Management: Additive (Backward-Compatible) Change

## Summary

A published event schema (`order_placed_v1`) is in production and consumed by multiple downstream systems including a Kafka consumer group, a Flink streaming job, a dbt bronze layer model, and a BI dashboard. The product team requires three new fields to be added to the schema: `referral_code` (optional string), `device_type` (optional enum), and `session_duration_ms` (optional integer with a default of `null`).

This response covers what makes a schema change backward compatible, how to update the data contract, perform consumer impact analysis, configure schema registry compatibility settings, and execute a safe version bump.

| Field | Type | Required | Default | Classification |
|---|---|---|---|---|
| `referral_code` | `string` | No | `null` | Additive, optional |
| `device_type` | `enum["web","mobile","api"]` | No | `null` | Additive, optional |
| `session_duration_ms` | `integer` | No | `null` | Additive, optional |

All three fields are **optional** with a `default` of `null`. This satisfies the definition of a backward-compatible change under both Avro and JSON Schema semantics.

---

## Decision

Proceed with a **backward compatible** minor version bump from `order_placed_v1` to `order_placed_v1.1.0` (semantic versioning). Register the updated schema in the schema registry with `BACKWARD` compatibility mode enforced. Existing consumers require **no code changes** and can be migrated to consume the new fields on their own schedule.

The data contract for the `order_placed` event will be updated to reflect the new fields, their types, optionality, and default values. The contract version will advance from `1.0.0` to `1.1.0`.

---

## Rationale

### What Makes This Change Backward Compatible

A schema change is **backward compatible** if and only if:
1. **New fields are optional** — existing producers that do not populate the field produce valid messages.
2. **New fields have a default** — consumers that read old messages (which lack the field) receive the default value rather than a deserialization error.
3. **No existing fields are removed** — consumers that reference existing fields continue to work.
4. **No existing field types are changed** — type coercion errors do not occur.

Adding `referral_code`, `device_type`, and `session_duration_ms` satisfies all four conditions.

### Avro Schema Delta

```json
// Before — order_placed v1.0.0 (excerpt)
{
  "type": "record",
  "name": "OrderPlaced",
  "namespace": "com.example.orders",
  "fields": [
    {"name": "order_id",    "type": "string"},
    {"name": "customer_id", "type": "string"},
    {"name": "order_total", "type": "double"},
    {"name": "created_at",  "type": "long", "logicalType": "timestamp-millis"}
  ]
}

// After — order_placed v1.1.0 (new optional fields appended)
{
  "type": "record",
  "name": "OrderPlaced",
  "namespace": "com.example.orders",
  "fields": [
    {"name": "order_id",             "type": "string"},
    {"name": "customer_id",          "type": "string"},
    {"name": "order_total",          "type": "double"},
    {"name": "created_at",           "type": "long", "logicalType": "timestamp-millis"},
    {"name": "referral_code",        "type": ["null", "string"],  "default": null},
    {"name": "device_type",          "type": ["null", "string"],  "default": null},
    {"name": "session_duration_ms",  "type": ["null", "long"],    "default": null}
  ]
}
```

Note: In Avro, `null` must appear first in the union when the `default` is `null`.

### Schema Registry Configuration

```bash
# Set compatibility level to BACKWARD for this subject
curl -X PUT \
  http://schema-registry:8081/config/orders.order_placed-value \
  -H "Content-Type: application/vnd.schemaregistry.v1+json" \
  -d '{"compatibility": "BACKWARD"}'

# Register the new schema version
curl -X POST \
  http://schema-registry:8081/subjects/orders.order_placed-value/versions \
  -H "Content-Type: application/vnd.schemaregistry.v1+json" \
  -d @schema_v1.1.0.json

# Verify the compatibility check passes
curl -X POST \
  http://schema-registry:8081/compatibility/subjects/orders.order_placed-value/versions/latest \
  -H "Content-Type: application/vnd.schemaregistry.v1+json" \
  -d @schema_v1.1.0.json
# Expected response: {"is_compatible":true}
```

Compatibility modes explained:
- `BACKWARD` — new schema can read data written by the old schema (consumers can upgrade first).
- `FORWARD` — old schema can read data written by the new schema (producers can upgrade first).
- `FULL` — both BACKWARD and FORWARD (safest; use for long-lived public schemas).

For this change, `BACKWARD` is sufficient. Consider upgrading to `FULL` if this schema is an external contract with SLA commitments.

### Data Contract Update

```yaml
# data_contract.yaml — version bump
schema:
  name: order_placed
  version: "1.1.0"          # bumped from 1.0.0
  format: avro
  compatibility: backward_compatible
  breaking_change: false

fields:
  - name: order_id
    type: string
    required: true
  - name: customer_id
    type: string
    required: true
  - name: order_total
    type: double
    required: true
  - name: created_at
    type: timestamp
    required: true
  # --- new fields (v1.1.0) ---
  - name: referral_code
    type: string
    required: false
    default: null
    added_in: "1.1.0"
    description: "Referral code applied at checkout, if any."
  - name: device_type
    type: string
    required: false
    default: null
    allowed_values: ["web", "mobile", "api"]
    added_in: "1.1.0"
    description: "Device category from which the order was placed."
  - name: session_duration_ms
    type: integer
    required: false
    default: null
    added_in: "1.1.0"
    description: "Duration of the shopping session in milliseconds."

consumers:
  - name: flink_order_enrichment
    team: streaming-platform
    impact: none
    migration_required: false
  - name: dbt_bronze_orders
    team: data-platform
    impact: low
    migration_required: false
  - name: analytics_dashboard
    team: bi-team
    impact: none
    migration_required: false
```

### Consumer Impact Analysis

| Consumer | Current Behavior | Impact of Change | Action Required |
|---|---|---|---|
| Flink `order_enrichment` job | Deserializes with schema v1.0.0 | Receives `null` for new fields; no error | None — update at team's discretion |
| dbt `stg_orders` bronze model | SELECT * from Kafka topic via connector | New columns surface as `NULL`; schema auto-evolves in BigQuery | Run `dbt run --select stg_orders` to pick up new columns |
| Analytics dashboard | Queries `mart_orders` | No new fields exposed yet | No action until mart model is updated |
| Data quality monitors | Row-count and null-rate checks on existing fields | Unaffected | Add null-rate monitors for new fields after rollout |

---

## Trade-offs

| Option | Pros | Cons |
|---|---|---|
| Minor version bump, optional fields (chosen) | Zero consumer downtime, no code changes required | New fields will be null-heavy initially; consumer adoption is voluntary and may lag |
| New topic `order_placed_v2` | Clean break, consumers can migrate at their own pace | Dual-topic overhead, producer must publish to both during transition |
| Embed new fields in a JSON `metadata` bag | No schema change needed | Loses type safety, breaks schema registry enforcement, makes field discovery opaque |
| Block change until all consumers acknowledge | Maximum safety | Blocks product delivery; impractical for async consumer teams |

**Default value choice:** Using `null` as the default (rather than a sentinel like `""` or `0`) is preferred because it preserves tri-state semantics (not collected vs. collected-but-empty vs. collected-with-value) and avoids accidental filtering in downstream `WHERE` clauses.

## Next Steps

1. **Merge the updated `data_contract.yaml`** to the schema repository. Trigger the contract validation CI job to confirm the `is_compatible: true` flag from the schema registry.

2. **Register schema v1.1.0** in the schema registry (staging first, then production) using the `curl` commands above. Confirm schema ID is returned and compatibility check passes.

3. **Notify all consumers** listed in the contract via a Slack post in `#data-platform-announcements` and a JIRA ticket per team. Message: "Schema `order_placed` has been updated to v1.1.0 with three new optional fields (`referral_code`, `device_type`, `session_duration_ms`). No action is required, but teams that want to use these fields can begin consuming them immediately."

4. **Update dbt bronze model** (`stg_orders`): add explicit column references for the three new fields with `COALESCE` defaults where appropriate. Run in CI against staging data before promoting.

5. **Add data quality monitors** for the new fields: null-rate baselines for the first 30 days, then alert if null rate drops below 10% unexpectedly (would indicate a producer bug sending garbage values).

6. **Document in the schema changelog**: Add a `CHANGELOG` entry with date, author, field names, rationale, and a link to the JIRA ticket.

7. **Schedule a 90-day review**: If any optional field has a null rate above 95% after 90 days, evaluate whether it should be removed in the next planned breaking-change window.
