# TC-E2E-019 — SPARK Mode: Structured Streaming Exactly-Once Guardrails

**Mode:** SPARK
**Scenario:** Reviewing a Spark Structured Streaming design for exactly-once correctness, covering checkpoints, watermarks, dedup, and idempotent sinks.
**Date Captured:** 2026-03-25

---

## Summary

A Spark Structured Streaming job ingests click events from a Kafka topic, enriches them with session metadata, and writes the output to a Delta Lake table for downstream analytics. A recent production incident revealed that after a job restart following an executor OOM crash, the job reprocessed approximately 45 minutes of Kafka offsets that had already been written to the Delta table. The duplicate rows caused downstream session-count metrics to be overstated by 18% for the affected window, requiring a manual backfill and correction run. The root cause was twofold: (1) the `checkpointLocation` was configured to local executor disk, which was lost when the executor was replaced, causing the job to restart from the beginning of the Kafka retention window; and (2) the Delta sink write used `append` mode with no dedup logic, so replayed events were simply appended as new rows.

Achieving exactly-once semantics in Spark Structured Streaming requires four components working together. First, the checkpoint must be stored in durable, replicated storage (HDFS, GCS, or S3) so that the job's committed Kafka offsets and operator state survive executor failures and container restarts. Second, a watermark must be defined on the event time column to bound the state maintained by stateful operators and to correctly handle late-arriving events without unbounded state growth. Third, explicit dedup logic using `dropDuplicates()` with the watermark column must be applied before the sink write to eliminate any duplicate event IDs that may arrive due to Kafka at-least-once delivery guarantees. Fourth, the sink must be idempotent — using Delta Lake's `MERGE` via `forEachBatch` ensures that re-processed batches update existing rows rather than inserting duplicates.

These four components form a defense-in-depth approach to exactly-once correctness. The checkpoint provides exactly-once offset tracking (source-side guarantee). The watermark scopes stateful dedup to a bounded time window, preventing unbounded memory growth. The `dropDuplicates()` call eliminates duplicates within the processing window. The idempotent Delta MERGE ensures that even if a batch is partially committed and then retried, the sink state converges to the correct result. No single component alone is sufficient — the checkpoint can survive a restart but cannot prevent duplicates from within-batch retries; the MERGE alone cannot prevent duplicates that arise from checkpoint loss if the dedup state is also lost.

---

## Decision

Redesign the streaming job with all four exactly-once guardrails in place:

1. **Move `checkpointLocation` to GCS** — Replace local disk path with a GCS bucket path (`gs://company-spark-checkpoints/click-events-job/`) to ensure checkpoint durability across executor failures and job restarts.

2. **Define a watermark on `event_time`** — Add `.withWatermark("event_time", "30 minutes")` to bound late-data state and enable stateful `dropDuplicates()` within a defined time window.

3. **Add `dropDuplicates()` before the sink** — Apply `.dropDuplicates(["event_id", "event_time"])` after the watermark definition to deduplicate within the watermark window.

4. **Use Delta MERGE in `forEachBatch` as the idempotent sink** — Replace the `append` mode write with a `forEachBatch` function that performs a Delta `MERGE` on `event_id`, updating existing rows and inserting new ones. This ensures re-processed batches produce no net change in the sink state.

## Rationale

**Checkpoint durability and idempotent restart.** The `checkpointLocation` is the single most critical configuration for a Structured Streaming job's fault tolerance. It stores three things: committed source offsets (Kafka topic/partition/offset tuples), operator state (for stateful operations like `dropDuplicates` and aggregations), and metadata about completed micro-batches. When the job restarts after a failure, it reads the checkpoint to determine where to resume Kafka consumption and what state to restore. If the checkpoint is on local disk and the disk is lost (executor replacement, node failure), the job has no committed offset record and defaults to the configured `startingOffsets` — often `latest` in production configs, which skips data, or `earliest`, which replays from the beginning of the Kafka retention window.

Storing the checkpoint on GCS (or S3/HDFS) ensures it persists through any single-node failure. The checkpoint path must be unique per job instance — sharing a checkpoint path between two concurrent job instances causes corruption. The pattern for uniquely pathing checkpoints is:

```python
checkpoint_path = "gs://company-spark-checkpoints/click-events-job/v1/"

query = (
    df
    .writeStream
    .format("delta")
    .outputMode("append")
    .option("checkpointLocation", checkpoint_path)
    .start(delta_table_path)
)
```

Note: changing the job's schema, watermark, or stateful operators may require clearing the checkpoint and restarting from scratch. Schema changes that are backward-compatible (adding nullable columns) can often be accommodated without checkpoint reset, but structural changes to stateful operators require a new checkpoint path.

**Watermark for late data and state expiry.** The watermark tells Spark the maximum expected lateness of event-time data. Events with `event_time` more than 30 minutes behind the current watermark are considered late and dropped. The watermark value advances as the job sees newer events — it is the maximum observed `event_time` minus the watermark delay. State maintained by `dropDuplicates()` and other stateful operators is only kept for the duration of the watermark window, then expired. Without a watermark, stateful operators accumulate state indefinitely, eventually exhausting executor memory.

```python
from pyspark.sql.functions import col

events_with_watermark = (
    raw_events
    .withWatermark("event_time", "30 minutes")
)
```

The 30-minute watermark is chosen to match the observed 99th percentile event delivery latency from Kafka to the Spark job. Events delivered more than 30 minutes late are acceptable to drop for this use case (click analytics). For use cases where late data must not be dropped, the watermark delay should be extended and the state store sized accordingly.

**Stateful dedup with `dropDuplicates()` and watermark.** `dropDuplicates()` in a streaming context maintains a state store of previously seen keys. With a watermark, it only retains keys for the duration of the watermark window — once a key's associated event time falls below the watermark threshold, it is evicted from the dedup state. This bounds memory usage at the cost of not deduplicating records that arrive after their watermark has expired. The dedup must specify the event time column alongside the unique key to enable time-bounded state eviction:

```python
deduped_events = (
    events_with_watermark
    .dropDuplicates(["event_id", "event_time"])
)
```

Using `dropDuplicates(["event_id"])` without `event_time` creates unbounded state — every `event_id` ever seen must be retained in state forever. Always include the event time column in the dedup key list when a watermark is defined.

**Idempotent Delta sink via `forEachBatch` and MERGE.** The `forEachBatch` function is called once per micro-batch with a static DataFrame representing that batch's output. By performing a Delta `MERGE` inside `forEachBatch`, each batch write becomes idempotent: if the same batch is retried (due to a transient write failure or speculative execution), the MERGE simply re-applies the same upserts and produces the same final state. The MERGE is keyed on `event_id` — if the event already exists in Delta, update its fields; if it does not exist, insert it.

```python
from delta.tables import DeltaTable

delta_table_path = "gs://company-data-lake/events/click_events"

def upsert_to_delta(batch_df, batch_id):
    delta_table = DeltaTable.forPath(spark, delta_table_path)
    (
        delta_table.alias("target")
        .merge(
            batch_df.alias("source"),
            "target.event_id = source.event_id"
        )
        .whenMatchedUpdateAll()
        .whenNotMatchedInsertAll()
        .execute()
    )

query = (
    deduped_events
    .writeStream
    .foreachBatch(upsert_to_delta)
    .option("checkpointLocation", checkpoint_path)
    .outputMode("update")
    .trigger(processingTime="1 minute")
    .start()
)
```

The `batch_id` parameter in `forEachBatch` is a monotonically increasing integer per micro-batch. It can be used to implement idempotency at the application level (e.g., skip re-processing if `batch_id` was already recorded in a control table), but Delta's `MERGE` makes this unnecessary for most use cases.

**End-to-end exactly-once guarantee.** With all four components in place, the exactly-once guarantee works as follows: the checkpoint records the Kafka offsets of the last successfully committed micro-batch. On restart, the job reads offsets from the checkpoint and resumes from exactly where it left off — no events are replayed from before the checkpoint, so the `dropDuplicates()` state (also restored from the checkpoint) is consistent. Any events that are replayed due to within-batch retry or speculative execution are caught by `dropDuplicates()`. Any duplicate writes that reach the sink despite dedup (e.g., due to a partial batch write before a checkpoint commit) are handled idempotently by the Delta MERGE. Each layer handles a different failure scenario.

## Trade-offs

| Approach | Exactly-Once? | State Bounded? | Sink Idempotent? | Complexity | Recovery Time |
|---|---|---|---|---|---|
| Append mode, no checkpoint (original) | No | Yes | No | Very Low | Full replay on restart |
| Durable checkpoint only, no dedup or MERGE | At-least-once | Yes | No | Low | Fast restart, duplicates possible |
| Checkpoint + watermark + dropDuplicates, append sink | At-least-once | Yes | No | Medium | Fast, rare duplicates on partial batch |
| Checkpoint + watermark + dropDuplicates + Delta MERGE (recommended) | Exactly-once (within watermark) | Yes | Yes | Medium-High | Fast restart, no duplicates |
| Checkpoint + global dedup (no watermark), Delta MERGE | Exactly-once | No — unbounded state | Yes | High | Fast, but OOM risk |
| Kafka transactions + idempotent producer only | At-least-once (Spark-side not covered) | N/A | Partial | Very High | Complex operational model |

**Exactly-once vs. at-least-once trade-offs.** Exactly-once semantics come with overhead: maintaining checkpoint state on durable storage adds latency to each micro-batch commit (typically 200–500ms for GCS checkpoint writes). The stateful `dropDuplicates()` operation requires a state store (RocksDB-backed by default) that consumes executor disk and memory. The Delta MERGE adds write amplification compared to a simple append. For most analytics use cases where 0.1% duplicate tolerance is acceptable, at-least-once with occasional dedup-on-read may be more cost-effective. For financial, billing, or compliance workloads, the overhead of exactly-once is justified and required.

**Watermark window size and late data.** A shorter watermark (e.g., 5 minutes) reduces state size and memory overhead but drops more late-arriving events. A longer watermark (e.g., 2 hours) captures more late arrivals but increases state store size, slows watermark advance, and increases end-to-end latency for aggregation results that wait for the watermark to pass before being finalized. The 30-minute watermark chosen here is a compromise based on observed delivery latency. It should be reviewed if the data source changes (e.g., a new mobile client with intermittent connectivity may have longer delivery latency).

**`forEachBatch` and micro-batch throughput.** Using `forEachBatch` with a Delta MERGE serializes sink writes — each micro-batch blocks until its MERGE completes before the next batch starts. For high-throughput topics, this can become a bottleneck if MERGE latency is high. Tuning the trigger interval (`processingTime`) and batch size (via Kafka `maxOffsetsPerTrigger`) is necessary to keep the MERGE within a target time budget. An alternative for very high throughput is to use Delta's native streaming write with `append` mode and schedule a periodic dedup compaction job — this sacrifices exactly-once for higher throughput.
