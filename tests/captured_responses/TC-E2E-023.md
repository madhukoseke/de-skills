# TC-E2E-023 — Schema Management: Breaking Change Migration (customer_id → account_id)

## Summary

The data platform team needs to rename the column `customer_id` to `account_id` across the `fact_orders` table and its upstream event schema. This change aligns the warehouse to the company's new canonical entity naming standard. However, `customer_id` is a public column referenced by downstream BI dashboards (Looker, Tableau), three dbt models, and two REST API consumers. Renaming it without a coordinated migration plan constitutes a **breaking change** that will silently corrupt BI reports and break API consumers if not managed carefully.

This response defines the full migration strategy: dual-write period, deprecation notice process, consumer migration plan, versioning strategy (`fact_orders_v2`), and rollback path.

---

## Decision

Classify this rename as a **breaking change** and execute a **dual-write migration** over a 6-week window. The strategy:

1. **Week 1**: Add `account_id` as a new column alongside the existing `customer_id` (dual-write begins). Issue deprecation notice to all consumers.
2. **Weeks 2–5**: Consumer migration window. Each team migrates to `account_id` on their own schedule. `customer_id` remains populated and authoritative during this period.
3. **Week 6**: Validate that no active consumers reference `customer_id`. Remove the column. Publish `fact_orders_v2` as the canonical table.

A **rollback** path is maintained throughout: `customer_id` is not dropped until all consumers have confirmed migration.

---

## Rationale

### Why This Is a Breaking Change

A **breaking change** is any schema modification that causes existing consumers to produce incorrect results or fail without code changes. Column renames are breaking because:
- SQL queries that reference `customer_id` by name will return `NULL` or raise an error after the rename.
- dbt models with explicit `SELECT customer_id` statements will fail compilation.
- BI tool dimensions keyed to `customer_id` will silently return NULL rows.
- API response serializers that map `customer_id` to a JSON field will break downstream JSON consumers.

Contrast with additive changes (adding an optional column): those are backward compatible. A rename is not — it is semantically equivalent to a delete + add.

### Dual-Write Period Design

During the dual-write window, both columns are populated simultaneously. The source-of-truth write path sets `account_id` first; `customer_id` is derived as an alias carrying the same value:

```sql
-- Migration step 1: add new column to fact_orders
ALTER TABLE warehouse.fact_orders
    ADD COLUMN account_id BIGINT;

-- Backfill historical rows
UPDATE warehouse.fact_orders
    SET account_id = customer_id
    WHERE account_id IS NULL;

-- Dual-write in the ETL pipeline
-- Both columns carry the same value during the migration window
account_id  = source.account_id,      -- canonical (new name)
customer_id = source.account_id       -- deprecated alias, same value
```

**Why populate `customer_id` with the new value?** In this rename, `customer_id` and `account_id` represent the same entity — the rename is cosmetic, not semantic. Both columns carry identical values during the dual-write window, ensuring consumers on either column see consistent data.

### Deprecation Notice Process

1. **Day 0 (Week 1, Monday)**: Publish a deprecation notice to all registered consumers via:
   - Slack `#data-platform-announcements`: "DEPRECATION NOTICE: `fact_orders.customer_id` will be removed on [date]. Migrate to `account_id`. See runbook: [link]."
   - JIRA tickets created in each consuming team's backlog.
   - Data contract `data_contract.yaml` updated with `deprecated_fields` section.
   - Column-level comment added in BigQuery: `ALTER TABLE fact_orders ALTER COLUMN customer_id SET OPTIONS (description='DEPRECATED: Use account_id. Will be removed YYYY-MM-DD.');`

2. **Week 3**: Mid-migration reminder. Query the warehouse audit log / query history to identify any active queries still referencing `customer_id`. Send targeted notices to those teams.

3. **Week 5**: Final notice. Any consumer still using `customer_id` must escalate to the data platform team for a migration extension or they accept breakage on the removal date.

### Versioning Strategy

```
fact_orders      → current table, dual-write window (both columns present)
fact_orders_v2   → post-migration table, only account_id present
```

`fact_orders_v2` is created as a **view** during the migration window, pointing to `fact_orders` but exposing only the new schema. This lets consumers that want to migrate early use `fact_orders_v2` immediately:

```sql
-- Create v2 view at week 1 for early adopters
CREATE OR REPLACE VIEW warehouse.fact_orders_v2 AS
SELECT
    order_key,
    account_id,           -- new canonical name
    -- customer_id intentionally excluded from the v2 surface
    order_date_key,
    product_key,
    order_total_usd,
    order_status,
    created_at
FROM warehouse.fact_orders;
```

After the migration window, `fact_orders` is updated to remove `customer_id` and `fact_orders_v2` is promoted to the primary table name.

### Detecting Active Usage Before Column Drop

```sql
-- BigQuery: scan INFORMATION_SCHEMA for recent queries referencing customer_id
SELECT
    user_email,
    query,
    creation_time
FROM `region-us`.INFORMATION_SCHEMA.JOBS_BY_PROJECT
WHERE
    LOWER(query) LIKE '%customer_id%'
    AND LOWER(query) LIKE '%fact_orders%'
    AND creation_time > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 7 DAY)
ORDER BY creation_time DESC;
```

Run this query in Week 5. Zero results is the go/no-go gate for column removal.

### Rollback Path

Rollback is possible at any point before the column is physically dropped:

1. **Rollback trigger**: A registered consumer reports breakage or a severity-1 incident is opened referencing `customer_id`.
2. **Immediate action**: Re-populate `customer_id` if any ETL step stopped writing to it. Since the column is never removed during the dual-write window, rollback is a no-op for consumers — `customer_id` continues to work.
3. **Cancel removal**: Remove the column-drop task from the migration runbook. Extend the dual-write window by 2 weeks.
4. **Post-incident review**: Identify the consumer not captured in the impact analysis. Update the consumer registry and data contract.

After the column is dropped, rollback requires:

```sql
-- Emergency rollback: restore customer_id from account_id
ALTER TABLE warehouse.fact_orders ADD COLUMN customer_id BIGINT;
UPDATE warehouse.fact_orders SET customer_id = account_id;
```

This operation may take 10–30 minutes on large tables. Plan a maintenance window if invoking a post-drop rollback.

## Trade-offs

| Approach | Pros | Cons |
|---|---|---|
| Dual-write 6-week window (chosen) | Zero-downtime migration, consumers set their own pace, clear rollback path | Operational overhead of maintaining two columns; schema bloat during transition |
| Immediate rename with advance notice | Simpler schema, no dual-write complexity | Any consumer that misses the deprecation notice breaks on cutover day; high blast radius |
| View aliasing only (no physical rename) | No consumer changes ever needed | Technical debt: the underlying column name stays wrong forever; confusing for new engineers |
| API versioning only (shim in API layer) | Isolates breakage to API layer | BI tools and dbt models still break; does not solve the warehouse-layer migration |

**Duration of dual-write window**: 6 weeks is a reasonable default for internal consumers. For external API consumers or SaaS customers, extend to 12 weeks and require formal written acknowledgment before proceeding with the column drop.

## Next Steps

1. **Update `data_contract.yaml`**: Add `deprecated_fields` entry for `customer_id` with removal date, and add `account_id` as the canonical replacement. Bump contract version from `1.x.x` to `2.0.0` to signal a breaking change.

2. **Execute Week 1 DDL**: Run the `ALTER TABLE ADD COLUMN account_id` and backfill script. Validate row counts before and after. Update the ETL pipeline to dual-write.

3. **Create `fact_orders_v2` view** immediately so early-adopter consumers can migrate before the window closes.

4. **File JIRA tickets** for each consumer team listed in the migration plan. Set target dates. Assign a data platform engineer as reviewer on each ticket to unblock questions.

5. **Schedule audit query** (Week 5) to detect any remaining `customer_id` references in query history. Make this a mandatory go/no-go gate before proceeding with the column drop.

6. **Execute column drop** (Week 6) only after the audit query returns zero results and all teams have confirmed migration. Drop the column in a maintenance window with the rollback SQL on standby.

7. **Post-migration cleanup**: Promote `fact_orders_v2` to `fact_orders`, drop the old view, update all documentation, and close the JIRA migration epic. Publish a completion notice to `#data-platform-announcements`.
