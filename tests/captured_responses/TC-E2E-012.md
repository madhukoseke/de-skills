## Summary
Define a data quality baseline for the orders fact table with freshness, completeness, uniqueness, and validity checks, plus explicit failure handling.

## Decision
Implement a DQ rule set that runs on every load: freshness SLA checks, completeness checks for required columns, uniqueness checks on business keys, and validity or referential integrity checks on constrained fields.

## Rationale
These checks catch the highest-frequency failure modes before bad data reaches downstream consumers. Pairing checks with explicit failure actions prevents silent data corruption and improves operational response time.

## Trade-offs
| Option | Pros | Cons | When to Use |
|--------|------|------|-------------|
| Fail on error + quarantine invalid rows | Strong data trust, clear containment | More pipeline interruptions | Critical marts and SLAs |
| Alert-only for all checks | Fewer hard failures | Risk of bad data reaching users | Early rollout or low-criticality datasets |

## Next Steps
