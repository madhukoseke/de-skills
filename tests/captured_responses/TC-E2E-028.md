# TC-E2E-028: Testing Strategy — Unit, Integration, and E2E for a New Pipeline

**Mode:** DATA_QUALITY / DBT
**Date:** 2026-03-25
**Scenario:** Designing a complete test strategy for a new dbt + Airflow pipeline using the DE testing pyramid.

---

## Summary

A new pipeline ingests raw clickstream events from Pub/Sub, transforms them through a three-layer dbt model (staging → intermediate → mart), and loads the mart into a dashboard-facing BigQuery dataset. This document defines the full testing pyramid: unit tests (dbt schema tests + PySpark unit tests with fixtures), integration tests (staging warehouse assertions, referential integrity), contract tests (schema validation against the data contract YAML), and e2e tests (full DAG run in staging with SLA and output assertions). Tool recommendations and go/no-go criteria are provided for each layer.

---

## Decision

**Adopt a four-layer testing pyramid** with the following distribution of test coverage:

| Layer | Test count target | Execution cadence | Gate |
|---|---|---|---|
| Unit test | 80+ (dbt schema + PySpark) | Every PR, <2 min | Blocks merge |
| Integration | 15–20 | Every PR to `main`, <10 min | Blocks merge |
| Contract test | 1 per producer interface | On schema change PR | Blocks merge |
| E2E | 3–5 critical paths | Nightly + pre-deploy | Blocks deploy |

---

## Rationale

### Layer 1: Unit Tests

Unit tests validate individual model logic and transformation functions in isolation, using controlled test data (fixtures) rather than production data.

#### dbt Schema Tests

dbt's built-in schema test framework provides fast, declarative unit test coverage for model outputs:

```yaml
# models/staging/schema.yml
models:
  - name: stg_clickstream_events
    columns:
      - name: event_id
        tests:
          - not_null
          - unique
      - name: user_id
        tests:
          - not_null
          - relationships:
              to: ref('stg_users')
              field: user_id
      - name: event_type
        tests:
          - accepted_values:
              values: ['page_view', 'click', 'conversion', 'session_start', 'session_end']
      - name: event_timestamp
        tests:
          - not_null
          - dbt_utils.expression_is_true:
              expression: "event_timestamp >= '2024-01-01'"
```

For custom business logic, use **dbt unit tests** (dbt-core >= 1.8):

```yaml
# models/marts/schema.yml — unit test for conversion rate calculation
unit_tests:
  - name: test_conversion_rate_calculation
    model: mart_user_conversion_funnel
    given:
      - input: ref('int_user_sessions')
        rows:
          - {user_id: 'u1', sessions: 10, conversions: 2}
          - {user_id: 'u2', sessions: 5, conversions: 0}
    expect:
      rows:
        - {user_id: 'u1', conversion_rate: 0.20}
        - {user_id: 'u2', conversion_rate: 0.00}
```

#### PySpark Unit Tests with Fixtures

For the ingestion preprocessing layer (PySpark), use `pytest` with `pyspark` in local mode and fixture-based test data. Fixtures eliminate production data dependencies and make tests deterministic:

```python
# tests/unit/test_event_transformer.py
import pytest
from pyspark.sql import SparkSession
from src.transformers.event_transformer import normalize_event_schema

@pytest.fixture(scope="session")
def spark():
    return SparkSession.builder.master("local[2]").appName("unit-tests").getOrCreate()

@pytest.fixture
def raw_events_fixture(spark):
    """Small, controlled test data — no production data used."""
    data = [
        {"event_id": "e1", "userId": "u1", "eventType": "page_view", "ts": 1700000000},
        {"event_id": "e2", "userId": None,  "eventType": "click",     "ts": 1700000001},
    ]
    return spark.createDataFrame(data)

def test_normalize_drops_null_user_id(spark, raw_events_fixture):
    result = normalize_event_schema(raw_events_fixture)
    assert result.filter(result.user_id.isNull()).count() == 0

def test_normalize_renames_columns(spark, raw_events_fixture):
    result = normalize_event_schema(raw_events_fixture)
    assert "user_id" in result.columns          # assert snake_case renaming applied
    assert "userId" not in result.columns
```

**Tools:** `pytest`, `pytest-spark`, `dbt test`, `dbt-unit-testing` (for dbt < 1.8).

### Layer 2: Integration Tests

Integration tests run dbt models against a **staging warehouse** (a separate BigQuery project or Snowflake dev environment) seeded with representative data, and assert on output properties including row counts and referential integrity:

```python
# tests/integration/test_mart_integrity.py
import pytest
from google.cloud import bigquery

@pytest.fixture(scope="module")
def bq_client():
    return bigquery.Client(project="my-project-staging")

def test_mart_row_count_nonzero(bq_client):
    """Assert that the mart loaded at least 1 row — catches silent empty loads."""
    query = "SELECT COUNT(*) AS cnt FROM `my-project-staging.marts.mart_user_conversion_funnel`"
    result = list(bq_client.query(query).result())[0]
    assert result.cnt > 0, "Mart is empty — dbt model produced no rows"

def test_no_duplicate_user_ids(bq_client):
    """Assert referential integrity — user_id must be unique in the mart."""
    query = """
        SELECT user_id, COUNT(*) AS cnt
        FROM `my-project-staging.marts.mart_user_conversion_funnel`
        GROUP BY user_id HAVING cnt > 1
    """
    duplicates = list(bq_client.query(query).result())
    assert len(duplicates) == 0, f"Duplicate user_ids found: {duplicates[:5]}"

def test_conversion_rate_in_valid_range(bq_client):
    """Assert business rule: conversion_rate must be between 0 and 1."""
    query = """
        SELECT COUNT(*) AS violations
        FROM `my-project-staging.marts.mart_user_conversion_funnel`
        WHERE conversion_rate < 0 OR conversion_rate > 1
    """
    result = list(bq_client.query(query).result())[0]
    assert result.violations == 0
```

**Tools:** `pytest`, `google-cloud-bigquery`, `dbt build --target staging`, `great_expectations` (for complex assertion sets).

### Layer 3: Contract Tests

Contract tests validate that the dbt model output schema exactly matches the published data contract YAML. This is a form of contract test that prevents silent breaking changes from reaching consumers:

```python
# tests/contract/test_schema_contract.py
import yaml
import pytest
from google.cloud import bigquery

def load_contract(path: str) -> dict:
    with open(path) as f:
        return yaml.safe_load(f)

@pytest.fixture
def contract():
    return load_contract("skills/data-engineering-best-practices/templates/data_contract.yaml")

def test_mart_schema_matches_contract(contract):
    """Contract test: assert every field in the data contract exists in the BigQuery table schema."""
    client = bigquery.Client(project="my-project-staging")
    table = client.get_table("my-project-staging.marts.mart_user_conversion_funnel")
    actual_fields = {f.name: f.field_type for f in table.schema}

    for field in contract["schema"]["fields"]:
        assert field["name"] in actual_fields, \
            f"Contract field '{field['name']}' missing from table schema"
        assert actual_fields[field["name"]].upper() == field["type"].upper(), \
            f"Type mismatch for '{field['name']}': contract={field['type']}, actual={actual_fields[field['name']]}"
```

A contract test failure on a schema-change PR is a hard block — the contract must be updated and re-versioned before the model change is merged.

**Tools:** `pyyaml`, `google-cloud-bigquery`, `soda-core` (for richer contract assertion DSL).

### Layer 4: E2E Tests

E2E tests trigger the full Airflow DAG in a staging environment and assert on end-to-end correctness — both the process (SLA) and the output (data matching expected fixtures):

```python
# tests/e2e/test_full_dag_run.py
import pytest
import time
from airflow.api.client.local_client import Client as AirflowClient
from datetime import datetime, timedelta

STAGING_DAG_ID = "clickstream_ingestion_pipeline"
SLA_MINUTES = 60  # Pipeline must complete within 60 minutes

def test_e2e_dag_completes_within_sla():
    """E2E: trigger the full DAG in staging and assert it completes within SLA."""
    client = AirflowClient(None, None)
    run_id = f"e2e_test_{datetime.utcnow().strftime('%Y%m%dT%H%M%S')}"

    client.trigger_dag(dag_id=STAGING_DAG_ID, run_id=run_id)

    deadline = datetime.utcnow() + timedelta(minutes=SLA_MINUTES)
    while datetime.utcnow() < deadline:
        state = client.get_dag_run(STAGING_DAG_ID, run_id)["state"]
        if state == "success":
            return  # PASS
        if state == "failed":
            pytest.fail(f"DAG run {run_id} failed before SLA deadline")
        time.sleep(30)

    pytest.fail(f"DAG run {run_id} did not complete within {SLA_MINUTES} minute SLA")

def test_e2e_output_matches_expected():
    """E2E: assert output row count and key metrics match expected fixture values."""
    from google.cloud import bigquery
    import json
    client = bigquery.Client(project="my-project-staging")

    with open("tests/fixtures/expected_mart_snapshot.json") as f:
        expected = json.load(f)

    query = "SELECT COUNT(*) AS cnt, SUM(conversions) AS total_conversions FROM `my-project-staging.marts.mart_user_conversion_funnel`"
    result = list(client.query(query).result())[0]

    assert abs(result.cnt - expected["row_count"]) / expected["row_count"] < 0.01, \
        "Row count differs from fixture by >1%"
    assert abs(result.total_conversions - expected["total_conversions"]) / expected["total_conversions"] < 0.05, \
        "Total conversions differs from fixture by >5%"
```

**Tools:** `apache-airflow` client, `pytest`, `google-cloud-bigquery`, custom fixture JSON files (stored under `tests/fixtures/`).

## Trade-offs

| Decision | Pros | Cons |
|---|---|---|
| dbt built-in schema tests for unit layer | Zero setup, fast, declarative | Only tests final column properties; cannot test intermediate transformation logic |
| PySpark local mode for unit tests | No cluster needed; fast CI execution | Does not catch distributed execution bugs (data skew, shuffle behavior) |
| Staging warehouse for integration/E2E | High fidelity; catches real warehouse-specific bugs | Costs money; requires managing staging data freshness and isolation between test runs |
| Contract tests blocking merge | Prevents silent breaking changes; enforces API discipline | Adds friction to schema evolution; teams must update contracts consciously (which is the point) |
| E2E tests as nightly + pre-deploy gate (not every PR) | Avoids 60-minute CI gate on every PR | A breaking change merged on Monday may not be caught until the nightly run |
| Fixture-based test data for unit tests | Deterministic; no prod data access required | Fixtures can drift from production patterns; must be maintained as schema evolves |

## Next Steps

1. **Sprint 1:** Implement the full dbt schema test suite (not_null, unique, accepted_values, relationships) for all three model layers. Run `dbt test --target staging` as a required CI check.
2. **Sprint 1:** Write PySpark unit tests with fixtures for the three core transformation functions. Target >80% line coverage on `src/transformers/`.
3. **Sprint 2:** Build the integration test suite (row count, referential integrity, business rule assertions). Wire into the PR-to-main CI pipeline.
4. **Sprint 2:** Implement contract tests against the data contract YAML. Add a schema-change detection step to the CI pipeline that triggers contract tests when `dbt/models/` or `templates/data_contract.yaml` changes.
5. **Sprint 3:** Deploy the E2E test suite to the staging Airflow environment. Configure nightly execution and pre-deploy gate in the CD pipeline.
6. **Ongoing:** Add test data fixture refresh to the staging pipeline (monthly snapshot from production with PII scrubbing) to keep integration and E2E fixtures representative of current production patterns.
