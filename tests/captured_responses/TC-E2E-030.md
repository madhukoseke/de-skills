# TC-E2E-030: DIAGNOSE — Post-Deploy Incident with Root Cause Analysis

**Mode:** DIAGNOSE
**Date:** 2026-03-25
**Incident ID:** INC-2026-0325-001
**Severity:** SEV-2 (downstream mart loads failing; BI dashboards showing stale data)

---

## Summary

At 14:32 UTC on 2026-03-24, a new dbt model (`int_user_sessions_v2`) was deployed to production as part of sprint release `v1.4.0`. At 16:05 UTC, automated monitoring detected failures in the downstream mart load tasks (`mart_user_conversion_funnel`, `mart_daily_active_users`). Investigation confirmed the new dbt model dropped the column `session_duration_seconds` without updating dependent downstream models. Immediate mitigation was achieved by rolling back the dbt model to the previous version at 17:22 UTC. Total blast radius: 4 mart tables, 11 dashboards, 3 downstream data consumers. This document provides the full timeline, blast radius analysis, root cause, mitigation steps, remediation plan, and a postmortem stub.

---

## Decision

**Immediate action:** Rollback `int_user_sessions_v2` to `int_user_sessions_v1` (previous tag: `v1.3.2`). Do not re-deploy until the column removal is handled via a safe deprecation pattern and `on_schema_change='fail'` is added to all dependent models.

**Root cause fix:** The column `session_duration_seconds` was removed from `int_user_sessions` without identifying or updating downstream consumers. The remediation requires: (1) retaining the column (or providing a backward-compatible alias), (2) adding `on_schema_change='fail'` to dependent models, (3) testing the full downstream graph in staging before re-deploying.

### Root Cause

**Primary root cause:** The dbt model `int_user_sessions_v2` removed the column `session_duration_seconds` as part of a schema refactor (the column was renamed to `avg_session_duration_ms` in the new version). No analysis was performed to identify downstream models that reference the old column name. dbt's `ref()` graph tracks model-to-model dependencies but does not automatically detect column-level lineage breaks unless `on_schema_change` is configured.

**Contributing factors:**

1. `on_schema_change` was not set on any downstream model configs — the default behavior (`'ignore'`) silently accepts schema changes without failing the build until runtime column references are evaluated.
2. The PR review for `int_user_sessions_v2` did not include a column lineage impact analysis.
3. Staging environment tests ran `dbt build --select int_user_sessions_v2` (only the changed model, not the full downstream graph), masking the downstream failures.

**Root cause statement:** A column was removed from a shared intermediate model without auditing downstream consumers, and the dbt configuration did not enforce schema change detection (`on_schema_change='fail'`), allowing the breaking change to reach production undetected.

### Immediate Mitigation — Rollback

The rollback was performed by checking out the `v1.3.2` git tag and running a targeted dbt run:

```bash
# Step 1: Check out previous release tag for the affected model
git checkout v1.3.2 -- models/intermediate/int_user_sessions.sql \
                        models/intermediate/schema.yml

# Step 2: Validate rollback in staging before applying to production
dbt build --select int_user_sessions+ --target staging

# Step 3: Deploy rollback to production
dbt run --select int_user_sessions+ --target prod

# Step 4: Trigger downstream mart refresh to restore current data
airflow dags trigger mart_refresh --exec-date 2026-03-24
```

The rollback restored `session_duration_seconds` to the intermediate model, allowing all four downstream mart models to build successfully.

### Remediation Steps (Permanent Fix)

Before re-introducing the `int_user_sessions_v2` changes, the following must be completed:

#### Step 1: Add `on_schema_change='fail'` to all downstream models

```yaml
# models/marts/schema.yml — add to each mart model config
models:
  - name: mart_user_conversion_funnel
    config:
      on_schema_change: "fail"    # Fail the build if upstream schema changes unexpectedly
  - name: mart_daily_active_users
    config:
      on_schema_change: "fail"
  # ... repeat for all mart models
```

#### Step 2: Handle the column rename with a backward-compatible alias

```sql
-- models/intermediate/int_user_sessions_v2.sql
SELECT
    session_id,
    user_id,
    avg_session_duration_ms,
    -- Backward-compatible alias: retain old column name during transition period
    avg_session_duration_ms / 1000.0 AS session_duration_seconds,
    ...
FROM {{ ref('stg_sessions') }}
```

#### Step 3: Update all downstream models to use the new column name

Update the four affected mart models to reference `avg_session_duration_ms` instead of `session_duration_seconds`. After updating all downstream models, remove the backward-compatible alias in a subsequent release.

#### Step 4: Test the full downstream graph in staging

```bash
# Run the full downstream graph — not just the changed model
dbt build --select int_user_sessions_v2+ --target staging
```

This ensures all downstream models are validated before production deployment.

## Trade-offs

| Decision | Pros | Cons |
|---|---|---|
| Immediate rollback vs. forward-fix | Fastest path to recovery; lower risk than deploying a hot fix under pressure | Loses the new features in `v1.4.0`; team must re-deploy after fixing |
| `on_schema_change='fail'` as default | Catches breaking schema changes at build time before runtime failures | Adds friction to intentional schema changes; engineers must explicitly handle schema evolution |
| Backward-compatible column alias during transition | Allows incremental migration of downstream models; zero-downtime column renames | Alias must be tracked and removed; adds technical debt if not cleaned up promptly |
| Narrow CI test scope (changed model only) vs. full downstream graph | Faster CI runs | Misses downstream breakage — as demonstrated by this incident. The CI time cost of running the full downstream graph is worth the reliability gain. |
