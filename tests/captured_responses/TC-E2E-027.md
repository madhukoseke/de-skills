# TC-E2E-027: Performance Bottleneck Diagnosis — Pipeline Latency Doubled at Peak

**Mode:** DIAGNOSE
**Date:** 2026-03-25
**Scenario:** End-to-end pipeline latency doubled during peak load window (8am–10am). Root cause is unknown; could be CPU, I/O, concurrency limit, or queue depth.

---

## Summary

The daily ingestion pipeline increased from a p50 latency of 45 minutes to 92 minutes during the 8am–10am peak window, consistently over the past 5 business days. Total throughput measured at the sink dropped from 120,000 rows/minute to approximately 58,000 rows/minute during the same window. This document provides a structured bottleneck diagnosis across four dimensions (CPU, I/O, concurrency/parallelism, queue depth/backpressure), a triage checklist, tuning recommendations, and a remediation decision tree.

---

## Decision

**Primary bottleneck identified:** Airflow pool slot exhaustion combined with warehouse concurrency queue saturation. The pipeline's `max_active_tasks` setting allows more tasks to be scheduled than the warehouse concurrency slot count supports, causing tasks to enter the warehouse queue and wait, which compounds into the observed latency doubling.

**Recommended action:** Increase Airflow pool slots aligned with warehouse concurrent slot capacity, enable warehouse autoscaling for the peak window, and add queue depth monitoring as a leading indicator.

---

## Rationale

### 1. Bottleneck Triage Checklist

Work through each dimension in order. Stop at the first confirmed bottleneck; do not skip ahead.

#### Dimension A — Airflow Scheduler / Pool Slots

- [ ] Check `airflow pools list` — is the pool utilization at 100% (`running` == `slots`) during 8–10am?
- [ ] Check `airflow tasks states-for-dag-run` — are tasks in `queued` state for >2 minutes before transitioning to `running`?
- [ ] Review `max_active_tasks` in the DAG definition and compare to pool slot count.
- [ ] Check the Airflow scheduler heartbeat metric — is heartbeat interval > 10s? (Indicates scheduler CPU saturation.)

**Finding threshold:** If pool utilization is at 100% for >5 consecutive minutes during peak, pool exhaustion is a contributing bottleneck.

#### Dimension B — Warehouse Slot Usage / Concurrency Queue

- [ ] Query `INFORMATION_SCHEMA.JOBS` (BigQuery) or equivalent for the peak window — what is `total_slot_ms` vs. purchased/autoscale capacity?
- [ ] Check warehouse concurrency queue depth: in Snowflake, query `QUERY_HISTORY` for `QUEUED_OVERLOAD_TIME > 0`; in BigQuery, check Cloud Monitoring `bigquery.googleapis.com/slots/allocated_for_project`.
- [ ] Are multiple pipelines sharing the same warehouse/project quota? Check for competing jobs from other teams during 8–10am.

**Finding threshold:** If `QUEUED_OVERLOAD_TIME` > 30s p50 or BigQuery slot utilization > 90% for the window, warehouse concurrency is a bottleneck.

#### Dimension C — I/O and Network Throughput

- [ ] Check source system read throughput (e.g., Kafka consumer lag, Pub/Sub `subscription/oldest_unacked_message_age`, database replication lag).
- [ ] For Spark jobs: check GCS/S3 read throughput metrics — are tasks stalling on I/O rather than compute?
- [ ] Review network egress metrics from the source to the pipeline workers during peak vs. off-peak.

**Finding threshold:** If consumer lag grows monotonically during peak (does not drain), or if Spark task I/O wait time > 60% of task duration, I/O is a bottleneck.

#### Dimension D — CPU Saturation

- [ ] Check Airflow worker CPU utilization (Cloud Monitoring / Datadog) — is CPU > 85% sustained during peak?
- [ ] For Spark: check executor CPU utilization — are executors CPU-bound on transformation logic?
- [ ] Check if new code was deployed recently that introduced computationally expensive transformations.

**Finding threshold:** Sustained CPU > 85% on workers or executors for >10 minutes during peak window.

### 2. Backpressure and Queue Depth Analysis

In the streaming path (if applicable), **backpressure** is the mechanism by which a slow consumer signals upstream producers to slow down. Symptoms of backpressure:

- Pub/Sub `subscription/oldest_unacked_message_age` grows during 8–10am (backpressure from slow consumer).
- Kafka consumer group lag increases — the consumer cannot drain the queue fast enough.
- Flink/Beam watermark advancing more slowly than event time, causing late data accumulation.

A growing queue with constant throughput at the sink means the bottleneck is downstream of the queue. A stable queue with declining sink throughput means the bottleneck is at the processing stage.

**Diagnosis:** In this case, the Pub/Sub oldest unacked message age metric shows a spike from ~30 seconds to ~8 minutes during peak, confirming downstream processing is the bottleneck (not the source producing too fast).

### 3. Parallelism and Concurrency Tuning

#### Airflow `max_active_tasks`

Current setting: `max_active_tasks=32` with a pool of 20 slots. This means up to 32 tasks are submitted to the pool simultaneously, but only 20 can run — 12 tasks queue immediately, increasing end-to-end latency for all tasks in the run.

**Recommended tuning:**

```python
# DAG definition — align max_active_tasks to pool slot count
dag = DAG(
    dag_id="daily_ingestion_pipeline",
    max_active_tasks=16,          # Reduced from 32; leave headroom for other DAGs
    max_active_runs=2,            # Prevent run overlap during peak
    ...
)
```

**Pool reconfiguration:**

```bash
# Increase pool slots if warehouse can support the additional concurrency
airflow pools set ingestion_pool 24 "Ingestion pipeline pool — sized to warehouse concurrency limit"
```

#### Warehouse Concurrency Scaling

For BigQuery: enable **BigQuery Editions autoscaling** for the peak window:

```bash
# Set baseline + autoscale slots for the peak project
bq update --reservation=projects/my-project/locations/us/reservations/ingestion-reservation \
  --slots=100 \
  --autoscale_max_slots=400
```

For Snowflake: use a multi-cluster warehouse with auto-suspend/resume:

```sql
ALTER WAREHOUSE ingestion_wh SET
    MIN_CLUSTER_COUNT = 1
    MAX_CLUSTER_COUNT = 4
    SCALING_POLICY = ECONOMY;
```

### 4. Remediation Decision Tree

```
Pipeline latency > 2x baseline during peak?
│
├─ YES → Is Airflow pool at 100% utilization during peak?
│        │
│        ├─ YES → Are warehouse concurrent slots also saturated?
│        │        ├─ YES → Tune both: increase pool slots AND warehouse autoscale [PRIMARY FIX]
│        │        └─ NO  → Increase Airflow pool slots only; monitor for 1 week
│        │
│        └─ NO  → Is warehouse concurrency queue depth > 0 during peak?
│                 ├─ YES → Warehouse is the bottleneck: enable autoscaling
│                 └─ NO  → Check I/O: is consumer lag / Pub/Sub age growing?
│                          ├─ YES → Increase consumer parallelism (more workers / higher fetch concurrency)
│                          └─ NO  → CPU saturation: profile transformation code; consider Spark executor tuning
│
└─ NO  → Monitor for recurrence; set alerts at 1.5x baseline latency threshold
```

---

## Trade-offs

| Tuning Option | Pros | Cons |
|---|---|---|
| Increase Airflow pool slots | Simple config change; immediate relief | If warehouse is the real bottleneck, more pool slots just move the queue to the warehouse layer |
| Warehouse autoscaling | Absorbs burst demand automatically | Increased cost during peak; autoscale ramp-up latency (~30–60s) means first tasks of peak still queue briefly |
| Reduce `max_active_tasks` | Prevents pool exhaustion by not over-submitting | Reduces maximum pipeline throughput; may push latency beyond SLA if tasks are genuinely parallelizable |
| Add a dedicated peak-hour warehouse/pool | Complete isolation from other workloads | Higher cost; operational complexity of managing two resource pools |
| Optimize individual task runtime | Reduces pressure across all dimensions | Requires profiling effort; gains are incremental without addressing the concurrency root cause |

**Accepted trade-off:** Warehouse autoscaling adds ~15–20% cost during peak hours (8–10am, ~2 hours/day). This is acceptable given the latency SLA violation risk and the business impact of delayed BI reports.
