## Summary
The table should use CREATE TABLE with partitioning on event date and clustering on customer_id to optimize pruning and cost for 50M rows/day.

## Decision
Use `CREATE TABLE` with `PARTITION BY event_date` and index/sort on `customer_id`. Estimate query cost using dry-run or EXPLAIN before production.

## Rationale
Partitioning is mandatory before clustering for large analytical tables. Requiring partition filters prevents accidental full scans and controls spend. Clustering by high-cardinality columns improves performance for common filters.

## Trade-offs
| Option | Pros | Cons | When to Use |
|--------|------|------|-------------|
| Partition + cluster | Best cost/performance balance | Requires modeling discipline | Large event tables |
| Cluster-only | Some locality benefits | Poor pruning and higher scan cost | Rarely; generally not recommended |
