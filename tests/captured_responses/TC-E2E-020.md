# TC-E2E-020 — DATA_MODELING Mode: SCD Type 2 Customer Dimension Design

**Mode:** DATA_MODELING
**Scenario:** Designing a customer dimension with full history tracking for `customer_tier` changes using Slowly Changing Dimension Type 2.
**Date Captured:** 2026-03-25

---

## Summary

The `customer_tier` attribute — which classifies customers into Bronze, Silver, Gold, or Platinum tiers based on annual spend — is a high-value dimension attribute that changes approximately 4% of the time per month as customers cross tier thresholds. The current data model uses a simple overwrite pattern (SCD Type 1): when a customer's tier changes, the existing row in `dim_customer` is updated in place, and the prior tier value is permanently lost. This has created a significant analytical blind spot: the revenue attribution team cannot determine what tier a customer held at the time of a historical transaction, making it impossible to accurately model tier-based revenue contribution, cohort retention by entry tier, or the causal impact of tier promotion on transaction frequency.

The solution is to convert the `customer_tier` dimension from SCD Type 1 to SCD Type 2. In a Type 2 design, each change to a tracked attribute generates a new row in the dimension table rather than overwriting the existing one. Three temporal validity columns are added: `valid_from` (the timestamp when this version of the row became active), `valid_to` (the timestamp when this version was superseded, or NULL/9999-12-31 for the current version), and `is_current` (a boolean flag for convenience, indicating the currently active version). A surrogate key — a synthetic, system-generated integer or UUID that uniquely identifies each dimension row — is introduced as the primary key of the dimension, distinct from the natural business key (`customer_id`). Fact tables join to the dimension using this surrogate key, which correctly resolves to the dimension version that was current at the time the fact event occurred.

The MERGE pattern for maintaining a Type 2 dimension is more complex than a Type 1 overwrite. When new data arrives, the MERGE must: (1) detect changed rows by comparing incoming `customer_tier` values against the current dimension version; (2) close the existing current row by setting `valid_to = CURRENT_TIMESTAMP` and `is_current = FALSE`; and (3) insert a new row with the new `customer_tier` value, `valid_from = CURRENT_TIMESTAMP`, `valid_to = NULL` (or `'9999-12-31'`), and `is_current = TRUE`. Rows where no tracked attribute has changed are left untouched. Rows that are new (no existing dimension record for this `customer_id`) are inserted directly as a first version with `is_current = TRUE`.

Fact table joins to the dimension must also be updated. Joins that require the current customer tier (e.g., for operational dashboards) use `WHERE is_current = TRUE`. Joins that require the historically accurate tier at the time of a transaction (e.g., for revenue attribution) join on `customer_id` AND `transaction_date BETWEEN valid_from AND COALESCE(valid_to, CURRENT_TIMESTAMP)`. This temporal join correctly resolves which dimension version was active at the time the fact event occurred, enabling accurate historical analysis that was previously impossible.

---

## Decision

Implement a full SCD Type 2 design for the `customer_tier` dimension with the following components:

1. **DDL with surrogate key, temporal validity columns, and `is_current` flag** — Create `dim_customer` with an auto-incremented surrogate key `customer_sk`, retain `customer_id` as the natural key, add `valid_from TIMESTAMP NOT NULL`, `valid_to TIMESTAMP` (nullable, NULL = current), `is_current BOOLEAN NOT NULL`, and track `customer_tier` as the slowly changing attribute.

2. **Idempotent MERGE pattern** — Implement a two-step MERGE: first, expire current rows where `customer_tier` has changed by setting `valid_to` and `is_current = FALSE`; second, insert new versions for changed customers and new customers.

3. **Fact table join pattern** — Document and implement both the current-snapshot join (`WHERE is_current = TRUE`) and the historical point-in-time join (`transaction_date BETWEEN valid_from AND COALESCE(valid_to, '9999-12-31')`) as standard patterns for all downstream consumers.

## Rationale

**DDL design with surrogate key and temporal validity.** The surrogate key is the cornerstone of a stable dimension table. Natural keys (`customer_id`) change infrequently but can be subject to system migrations, merges, or corrections that make them unreliable as foreign keys in fact tables. Surrogate keys are system-assigned, never reused, and completely stable. Fact tables store the `customer_sk` value at the time the event occurred; even if the natural key is later corrected, the fact table's foreign key still points to the correct historical dimension version.

```sql
CREATE TABLE dim_customer (
    -- Surrogate key: uniquely identifies each dimension row (version)
    customer_sk         BIGINT          NOT NULL GENERATED ALWAYS AS IDENTITY PRIMARY KEY,

    -- Natural key: business identifier, non-unique in this table (multiple versions per customer)
    customer_id         VARCHAR(64)     NOT NULL,

    -- Slowly changing attributes tracked via SCD Type 2
    customer_tier       VARCHAR(20)     NOT NULL,  -- 'Bronze', 'Silver', 'Gold', 'Platinum'

    -- Non-tracked attributes (SCD Type 1 overwrite within current row)
    customer_name       VARCHAR(255)    NOT NULL,
    email               VARCHAR(255),
    country_code        CHAR(2),

    -- Temporal validity columns
    valid_from          TIMESTAMP       NOT NULL DEFAULT CURRENT_TIMESTAMP,
    valid_to            TIMESTAMP,      -- NULL means this is the current (active) version
    is_current          BOOLEAN         NOT NULL DEFAULT TRUE,

    -- Audit columns
    created_at          TIMESTAMP       NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at          TIMESTAMP       NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- Index for natural key lookup (e.g., looking up current record by customer_id)
CREATE INDEX idx_dim_customer_natural_key ON dim_customer (customer_id, is_current);

-- Index for historical join (temporal range query)
CREATE INDEX idx_dim_customer_temporal ON dim_customer (customer_id, valid_from, valid_to);
```

**MERGE pattern to close old rows and insert new versions.** The SCD Type 2 MERGE is a two-step operation. Most SQL dialects (including Snowflake, BigQuery, and Databricks SQL) allow both steps within a single MERGE statement using multiple WHEN MATCHED conditions. In databases that do not support multi-action MERGE, use explicit UPDATE then INSERT statements within a transaction.

```sql
-- Step 1: Expire current rows where customer_tier has changed
MERGE INTO dim_customer AS target
USING (
    -- Source: incoming customer records from staging
    SELECT
        customer_id,
        customer_tier,
        customer_name,
        email,
        country_code
    FROM stg_customers
) AS source
ON target.customer_id = source.customer_id
   AND target.is_current = TRUE

-- Close the existing current row if the tracked attribute has changed
WHEN MATCHED AND target.customer_tier <> source.customer_tier THEN
    UPDATE SET
        target.valid_to    = CURRENT_TIMESTAMP,
        target.is_current  = FALSE,
        target.updated_at  = CURRENT_TIMESTAMP

-- Update non-tracked attributes in place for unchanged tier (SCD Type 1 behavior for non-tracked fields)
WHEN MATCHED AND target.customer_tier = source.customer_tier THEN
    UPDATE SET
        target.customer_name = source.customer_name,
        target.email         = source.email,
        target.country_code  = source.country_code,
        target.updated_at    = CURRENT_TIMESTAMP;

-- Step 2: Insert new dimension rows for changed customers and new customers
INSERT INTO dim_customer (
    customer_id, customer_tier, customer_name, email, country_code,
    valid_from, valid_to, is_current
)
SELECT
    s.customer_id,
    s.customer_tier,
    s.customer_name,
    s.email,
    s.country_code,
    CURRENT_TIMESTAMP AS valid_from,
    NULL              AS valid_to,
    TRUE              AS is_current
FROM stg_customers s
-- Insert if the customer is new (no existing dimension row)
WHERE NOT EXISTS (
    SELECT 1 FROM dim_customer d WHERE d.customer_id = s.customer_id AND d.is_current = TRUE
)
-- Also insert if the customer's tier changed (the old row was just closed above)
UNION ALL
SELECT
    s.customer_id,
    s.customer_tier,
    s.customer_name,
    s.email,
    s.country_code,
    CURRENT_TIMESTAMP AS valid_from,
    NULL              AS valid_to,
    TRUE              AS is_current
FROM stg_customers s
JOIN dim_customer d
  ON d.customer_id = s.customer_id
 AND d.is_current = FALSE
 AND d.valid_to >= CURRENT_TIMESTAMP - INTERVAL '1 minute'  -- just expired in this run
WHERE d.customer_tier <> s.customer_tier;
```

In Delta Lake (Databricks), the SCD Type 2 MERGE can be expressed more cleanly using the `MERGE` with `whenMatchedUpdate` and `whenNotMatchedInsert` API, with the delta table's `OPTIMIZE` and `ZORDER BY customer_id` applied afterwards for read performance.

**Fact table joins: current vs. historical.** Two join patterns are required for different analytical use cases:

Current-snapshot join (for operational dashboards showing today's state):
```sql
SELECT
    f.transaction_id,
    f.transaction_date,
    f.revenue,
    d.customer_tier,
    d.customer_name
FROM fact_transactions f
JOIN dim_customer d
  ON f.customer_sk = d.customer_sk
WHERE d.is_current = TRUE;
```

Historical point-in-time join (for accurate revenue attribution at time of transaction):
```sql
-- Using the surrogate key stored at fact-load time (preferred — no range scan needed)
SELECT
    f.transaction_id,
    f.transaction_date,
    f.revenue,
    d.customer_tier   AS tier_at_transaction_time,
    d.customer_name
FROM fact_transactions f
JOIN dim_customer d
  ON f.customer_sk = d.customer_sk;
-- No WHERE clause needed: the surrogate key already identifies the exact dimension version

-- Alternative: range-based temporal join (when customer_sk is not stored in fact table)
SELECT
    f.transaction_id,
    f.transaction_date,
    f.revenue,
    d.customer_tier   AS tier_at_transaction_time
FROM fact_transactions f
JOIN dim_customer d
  ON f.customer_id = d.customer_id
 AND f.transaction_date >= d.valid_from
 AND f.transaction_date < COALESCE(d.valid_to, '9999-12-31');
```

The surrogate-key approach (first query) is strongly preferred because it avoids the range scan on the temporal join columns. The surrogate key is resolved at fact-load time and stored in the fact table; the join at query time is a simple equality join, which is efficient and optimizer-friendly. The range-based temporal join (second query) is a fallback for legacy fact tables that do not store `customer_sk`.

**Handling the `is_current` flag consistency.** The `is_current = TRUE` flag must be consistent with the `valid_to IS NULL` condition — exactly one row per `customer_id` should have `is_current = TRUE` and `valid_to IS NULL` at any given time. A data quality check should be run after each dimension load to verify this invariant:

```sql
-- Assert: no customer has more than one current row
SELECT customer_id, COUNT(*) AS current_row_count
FROM dim_customer
WHERE is_current = TRUE
GROUP BY customer_id
HAVING COUNT(*) > 1;
-- Should return 0 rows
```

## Trade-offs

| SCD Type | History Preserved | Storage Cost | ETL Complexity | Query Complexity | Best For |
|---|---|---|---|---|---|
| Type 1 (overwrite) | No | Low | Very Low | Very Low | Non-analytical reference data, current-state only |
| Type 2 (add row, recommended) | Full history | Medium (1.04x with 4% monthly change rate) | Medium-High | Medium | Tier, status, segment changes needing point-in-time accuracy |
| Type 3 (add column for prior value) | One prior value only | Low | Low | Low | Simple before/after comparison, no deep history needed |
| Type 4 (separate history table) | Full history | Medium | Medium | Medium-High | Very large dimensions where version rows would bloat the main table |
| Type 6 (hybrid 1+2+3) | Full history + current + prior | High | Very High | Medium | Complex analytical needs requiring multiple access patterns |

**SCD Type 1 vs. Type 2.** Type 1 is appropriate when the attribute's historical value has no analytical significance — for example, correcting a typo in a customer's email address. Type 2 is appropriate when the attribute's value at the time of a historical event is analytically meaningful — for example, a customer's tier at the time of a purchase. The key question is: "If this attribute changed, would the meaning of a historical fact row be different depending on which value of this attribute we use?" For `customer_tier`, the answer is yes: a $500 purchase made when the customer was Gold tier has different analytical significance than the same purchase made when they were Bronze tier.

**Storage overhead of Type 2.** With a 4% monthly change rate, each customer generates approximately 0.52 new dimension versions per year on average (some customers change multiple times). Over 5 years, the average customer accumulates ~2.6 dimension rows. The `dim_customer` table will therefore be approximately 2.6x its original row count after 5 years, compared to a Type 1 table which stays constant. At 50 million customers and ~500 bytes per row, this is approximately 65 GB — a manageable overhead for the analytical value gained.

**Temporal join complexity and performance.** Range-based temporal joins (`transaction_date BETWEEN valid_from AND valid_to`) are significantly more expensive than equality joins (`customer_sk = customer_sk`) because they cannot use standard B-tree index equality lookups. The strongly recommended mitigation is to resolve the surrogate key at fact-load time and store it in the fact table. This converts the query-time join from a range scan to an equality join, eliminating the performance penalty. Teams that cannot modify the fact table ETL to store `customer_sk` should ensure that `(customer_id, valid_from, valid_to)` has a composite index and that query engines can use nested loop + index scan strategies for the temporal join.

**`valid_to NULL` vs. `valid_to = '9999-12-31'`.** Using `NULL` for the open-ended `valid_to` is semantically correct (it means "no end date") but complicates range comparisons because `NULL` comparisons require `IS NULL` or `COALESCE`. Using the sentinel value `'9999-12-31'` makes range queries simpler (`transaction_date < valid_to` works without COALESCE) but requires special handling in downstream tools that may not expect future dates. Both are widely used in practice; the sentinel value approach is slightly more query-friendly and is recommended here for consistency with most BI tool expectations.

## Next Steps

1. **Implement DDL and backfill historical dimension** — Create the new `dim_customer` table with the SCD Type 2 schema. Run a one-time backfill using the transaction history and any available customer tier change logs to populate historical dimension versions. Where historical change timestamps are not available, use `transaction_date` as a proxy for `valid_from`. Assign surrogate keys (`customer_sk`) to each historical version. Owner: data modeling team. Target: 3 weeks.

2. **Implement and test the MERGE-based dimension load job** — Build the two-step MERGE ETL job in the orchestration platform (Airflow). Add unit tests for: (a) new customer insert, (b) tier change closes old row and inserts new, (c) non-tracked attribute update in place, (d) idempotent re-run produces no change. Run in staging on 6 months of data before production deployment.

3. **Update fact table ETL to store `customer_sk`** — Modify the `fact_transactions` ETL job to resolve `customer_sk` at load time by looking up `WHERE customer_id = ? AND is_current = TRUE`. This eliminates the need for range-based temporal joins at query time and is the most important performance optimization for the new dimension design.

4. **Update downstream dbt models** — Audit all dbt models that reference `dim_customer` and update their join logic to: (a) use `WHERE is_current = TRUE` for current-state reporting, and (b) join via `customer_sk` for historical accuracy. Add a dbt test to assert that no model joins `fact_transactions` to `dim_customer` on `customer_id` alone (without temporal scoping), as this produces fan-out duplicates.

5. **Establish data quality checks for SCD invariants** — Add automated checks to the dimension load job: (1) no `customer_id` has more than one `is_current = TRUE` row; (2) no row has `valid_from > valid_to` (where `valid_to IS NOT NULL`); (3) for each `customer_id`, dimension versions are non-overlapping and contiguous (no gaps between `valid_to` of one version and `valid_from` of the next). Fail the load job if any check fails.

6. **Document the point-in-time join pattern for analytics users** — Write a short guide for the analytics and data science teams explaining SCD Type 2 concepts, the two join patterns (current vs. historical), and worked examples in SQL. Publish to the data catalog / analytics wiki. This prevents analysts from inadvertently joining on `customer_id` alone and producing incorrect historical results.
