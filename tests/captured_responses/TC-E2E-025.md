# TC-E2E-025 — Deploy Strategy: 90-Day Historical Backfill

## Summary

A new pipeline (`pipeline_order_events_v2`) has been built to process `order_events` data with an improved enrichment layer. Before routing live traffic to it, the team must backfill 90 days of historical data from the source system. Naively loading 90 days in a single job would overwhelm the data warehouse (slot contention, cost spikes, lock timeouts) and produce a non-idempotent load that is unsafe to retry. This response defines a **windowed backfill** strategy with idempotent re-run design, Airflow throttling controls, a reconciliation/checksum step, and the correct order of operations before live traffic cutover.

---

## Decision

Execute the backfill as a series of **daily windows**, one batch per calendar day, processed over 90 Airflow DAG runs with `max_active_runs=3` for controlled parallelism. Each window is independently **idempotent** — safe to retry without duplicating data. Airflow `pool` and `max_active_runs` limits enforce throttling to prevent warehouse overload. A **reconciliation** step validates row counts and checksums against the source system before live traffic is cut over.

**Order of operations:**
```
1. Backfill (90 daily windows, oldest-first, max 3 concurrent)
2. Reconciliation / checksum validation per window
3. Go/no-go decision gate (all windows pass reconciliation)
4. Cutover: route live traffic to new pipeline
5. Monitor for 24 hours post-cutover
6. Decommission legacy pipeline
```

---

## Rationale

### Windowed Backfill Strategy

Processing 90 days as a single batch is unsafe because:
- Large single-query scans cause warehouse slot exhaustion, blocking other production queries.
- A failure at day 89 requires reprocessing all 90 days unless checkpoints are implemented.
- Write operations on large datasets hold table locks longer, increasing contention.

A **window**-based approach processes one calendar day per run. Each window is small enough to complete within a predictable time budget (target: < 15 minutes per day-window), and failures are isolated to a single window rather than the entire backfill.

```python
# Airflow DAG — windowed backfill
from datetime import datetime, timedelta
from airflow import DAG
from airflow.operators.python import PythonOperator

BACKFILL_START = datetime(2025, 12, 26)   # 90 days before 2026-03-25
BACKFILL_END   = datetime(2026, 3, 24)

dag = DAG(
    dag_id="pipeline_order_events_v2_backfill",
    start_date=BACKFILL_START,
    end_date=BACKFILL_END,
    schedule_interval="@daily",
    max_active_runs=3,      # throttle: process max 3 day-windows concurrently
    catchup=True,           # process all historical dates
    default_args={
        "retries": 3,
        "retry_delay": timedelta(minutes=5),
    },
)
```

### Idempotent Re-run Design

Each window uses a **DELETE + INSERT** (or partition-overwrite) pattern scoped to the window's date partition. This ensures the window is safe to retry at any time: a re-run first deletes any partially written data for that date, then re-inserts clean data.

```sql
-- Idempotent window load: DELETE then INSERT within a transaction
BEGIN TRANSACTION;

DELETE FROM warehouse.fact_order_events
WHERE event_date = '{{ ds }}';

INSERT INTO warehouse.fact_order_events
SELECT
    event_id,
    order_id,
    account_id,
    event_type,
    event_timestamp,
    DATE(event_timestamp)   AS event_date,
    payload_json,
    CURRENT_TIMESTAMP()     AS dw_loaded_at
FROM staging.stg_order_events_raw
WHERE DATE(event_timestamp) = '{{ ds }}'
  AND _source_system = 'order_service';

COMMIT;
```

For BigQuery (which uses partition decorators):

```python
def load_window(ds: str, **context):
    """Idempotent: WRITE_TRUNCATE overwrites the partition for ds safely."""
    client = bigquery.Client()
    job_config = bigquery.QueryJobConfig(
        destination=f"project.warehouse.fact_order_events${ds.replace('-','')}",
        write_disposition=bigquery.WriteDisposition.WRITE_TRUNCATE,  # idempotent
        time_partitioning=bigquery.TimePartitioning(field="event_date"),
        create_disposition=bigquery.CreateDisposition.CREATE_IF_NEEDED,
    )
    query = f"""
        SELECT * FROM staging.stg_order_events_raw
        WHERE DATE(event_timestamp) = '{ds}'
    """
    job = client.query(query, job_config=job_config)
    job.result()
    return job.output_rows
```

`WRITE_TRUNCATE` on a partition decorator makes each window idempotent: re-running overwrites the same partition, producing identical results regardless of how many times it is retried.

### Throttling and Concurrency Controls

**Airflow pool for backfill isolation:**

```bash
# Create a dedicated Airflow pool for backfill jobs
# Limits total concurrent backfill tasks to 5 across all DAGs
airflow pools set backfill_pool 5 "Dedicated pool for historical backfill jobs"
```

```python
load_task = PythonOperator(
    task_id="load_window",
    python_callable=load_window,
    pool="backfill_pool",        # dedicated pool prevents starvation of production tasks
    pool_slots=1,
    dag=dag,
)
```

**Why `max_active_runs=3`?**

Running windows strictly sequentially (1 at a time) would take 90 × 15 minutes ≈ 22.5 hours. Running 3 in parallel reduces wall-clock time to approximately 7.5 hours while keeping warehouse load manageable. Running more than 5 in parallel risks slot contention with production workloads during business hours. The `backfill_pool` cap of 5 slots enforces this ceiling across all active windows.

**Warehouse-level throttle (BigQuery reservation):**

```python

)
```

### Reconciliation and Checksum Validation

After all 90 windows complete, a mandatory **reconciliation** step runs before cutover. It compares row counts and a deterministic checksum between the source system and the warehouse for every window:

```python
def reconcile_backfill(**context):
    """
    Compare row counts and checksums for each backfill window.
    Raises a ValueError (blocking cutover) if any window fails.
    """
    client = bigquery.Client()
    source_client = get_source_db_client()

    failed = []
    for ds in date_range(BACKFILL_START, BACKFILL_END):
        source_count = source_client.query(
            f"SELECT COUNT(*) FROM order_events WHERE DATE(event_timestamp) = '{ds}'"
        ).scalar()

        wh_count = client.query(
            f"SELECT COUNT(*) FROM warehouse.fact_order_events WHERE event_date = '{ds}'"
        ).scalar()

        # Checksum: BIT_XOR of fingerprints is order-independent and fast
        source_checksum = source_client.query(
            f"""SELECT BIT_XOR(CAST(FARM_FINGERPRINT(event_id) AS INT64))
                FROM order_events WHERE DATE(event_timestamp) = '{ds}'"""
        ).scalar()

        wh_checksum = client.query(
            f"""SELECT BIT_XOR(CAST(FARM_FINGERPRINT(event_id) AS INT64))
                FROM warehouse.fact_order_events WHERE event_date = '{ds}'"""
        ).scalar()

        row_match      = abs(source_count - wh_count) / max(source_count, 1) < 0.001
        checksum_match = (source_checksum == wh_checksum)

        if not row_match or not checksum_match:
            failed.append({"date": ds, "source": source_count, "wh": wh_count})
            trigger_window_rerun(ds)   # re-queue failed window automatically

    if failed:
        raise ValueError(
            f"Reconciliation failed for {len(failed)} windows: "
            f"{[r['date'] for r in failed]}. Cutover is blocked."
        )

    logging.info("All 90 windows passed reconciliation. Cutover approved.")
```

**Why checksum in addition to row count?** Row count reconciliation alone can miss errors where rows are deleted and re-inserted with different values (net count is correct, but data is wrong). The BIT_XOR checksum catches value-level corruption at the cost of one additional query per window per system — a trade-off that is strongly worth taking for a production backfill.

**Reconciliation tolerance**: A row count tolerance of 0.1% accounts for late-arriving records written to the source database after the backfill window was processed. Any window outside this tolerance triggers an automatic window re-run before cutover proceeds.

---

## Trade-offs

| Decision | Pros | Cons |
|---|---|---|
| Daily windows, `max_active_runs=3` (chosen) | Predictable load, isolated failures, fast retry per window | ~7.5 hours total wall-clock time; requires careful pool sizing |
| Weekly windows | Fewer Airflow task instances | Larger blast radius per failure; wider reconciliation gaps; longer individual task runtime |
| Single-batch load | Simplest implementation | Unrecoverable on failure without full restart; likely causes warehouse overload |
| Streaming backfill (replay Kafka from offset) | Near-real-time; no batch window design needed | Requires 90-day Kafka retention (unusual); replay speed must be throttled separately |
| `WRITE_TRUNCATE` per partition (chosen for BigQuery) | Idempotent by design; partition isolation | Requires partitioned table; cannot use on non-partitioned tables |
| MERGE (UPSERT) for idempotency | Handles late-arriving updates gracefully | Slower than WRITE_TRUNCATE; more complex query plan; requires a unique key |

**Checksum vs. row count only**: The BIT_XOR approach is order-independent — it does not require rows to be in the same sequence in source and warehouse — and is computationally cheap even for hundreds of millions of rows. The cost is one extra query per window; the benefit is catching data corruption that row counts alone would miss.

**`max_active_runs` tuning**: 3 is conservative. If the backfill needs to complete faster (e.g., a hard cutover deadline), increase to 5 and reduce the BigQuery reservation slot budget proportionally to maintain production SLO headroom.

## Next Steps

1. **Provision the Airflow pool**: Run `airflow pools set backfill_pool 5 "Backfill pool"` in production before triggering the DAG. Confirm the pool appears in the Airflow UI with 5 slots available.

2. **Dry-run a single window** (e.g., the most recent day) in staging. Verify idempotency by running it twice and confirming row counts are identical on both runs. Verify the checksum matches the source.

3. **Trigger the backfill DAG** during off-peak hours (e.g., Friday evening) with `start_date=BACKFILL_START` and `catchup=True` to minimize competition with production workloads.

4. **Monitor throughout the backfill**: Watch Airflow pool utilization, BigQuery slot usage, and per-window task durations. Alert if average window duration exceeds 20 minutes, which indicates warehouse contention.

5. **Run reconciliation** after all 90 windows report success. Review the report for any windows with row count delta > 0.1%. Re-run failed windows and re-reconcile until all windows pass.

6. **Execute cutover** during a low-risk window with an on-call engineer available. Pause the legacy pipeline, unpause `pipeline_order_events_v2` for live traffic, and confirm the first live-traffic run produces correct output.

7. **Post-backfill cleanup**: After a 7-day stabilization period, drop the backfill DAG and any temporary staging tables. Document the backfill approach — window size, pool configuration, reconciliation tolerance — in the pipeline runbook for future reference.
