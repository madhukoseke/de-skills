# TC-E2E-026: Performance Optimization for a Slow, Expensive Warehouse Query

**Mode:** WAREHOUSE
**Date:** 2026-03-25
**Scenario:** A BI dashboard query full-scans a 5TB table with no partition filter, uses a subquery instead of a JOIN, and runs 50 times per day.

---

## Summary

A BI dashboard query against `analytics.events` is consuming approximately $1,562.50/day in BigQuery compute costs due to three compounding issues: no partition filter (full 5TB scan per execution), a correlated subquery that prevents the query optimizer from pushing down predicates, and a high-frequency execution pattern (50 runs/day) with no result caching. This review covers EXPLAIN plan analysis, partition pruning, query rewrite, cost modeling, and a recommendation to introduce a materialized view for the repeated access pattern.

---

## Decision

**Recommended:** Implement partition filter pruning, rewrite the correlated subquery as a JOIN, and create a materialized view refreshed on a scheduled cadence to serve BI dashboard reads.

**Rejected alternative:** Result cache only — caching alone does not reduce bytes scanned on cache misses and provides no cost protection on cold cache scenarios.

---

## Rationale

### 1. EXPLAIN Plan Analysis

Running `EXPLAIN` on the current query surfaces the following:

```sql
-- Current query (problematic)
SELECT
    u.user_id,
    u.email,
    (SELECT COUNT(*) FROM analytics.events e WHERE e.user_id = u.user_id AND e.event_date >= '2026-01-01') AS event_count
FROM analytics.users u
WHERE u.created_at >= '2026-01-01';
```

The `EXPLAIN` output shows:
- **Full table scan** on `analytics.events` (5TB): no partition pruning — `event_date` partition column is inside the subquery, which the optimizer cannot hoist.
- **Correlated subquery**: executes once per row in `users`, producing an O(n) scan pattern.
- **Estimated bytes processed:** 5,000 GB per execution.
- **No index** (clustered column) utilization on `user_id` — the subquery prevents the clustering index on `events` from being used effectively.

### 2. Adding a Partition Filter

`analytics.events` is partitioned on `event_date`. Adding a partition filter in the outer query context and rewriting as a JOIN allows BigQuery to apply **partition pruning**:

```sql
-- After fix: partition filter applied, subquery rewritten as JOIN
SELECT
    u.user_id,
    u.email,
    COUNT(e.event_id) AS event_count
FROM analytics.users u
LEFT JOIN analytics.events e
    ON e.user_id = u.user_id
   AND e.event_date >= '2026-01-01'      -- partition filter: now applied to partitioned column
WHERE u.created_at >= '2026-01-01'
GROUP BY u.user_id, u.email;
```

With the partition filter on `event_date >= '2026-01-01'`, only ~50GB of the 5TB table is scanned (approximately the last 90 days of data out of 5 years). The clustering index on `user_id` within those partitions is now also leveraged.

### 3. Cost Savings Estimate

BigQuery on-demand pricing: **$6.25 per TB scanned**. Bytes scanned directly determine cost.

| Scenario | Bytes scanned per run | Cost per run | Daily cost (50 runs) |
|---|---|---|---|
| Before (no partition filter) | 5,000 GB (5 TB) | $31.25 | $1,562.50 |
| After (partition filter + JOIN) | ~50 GB (0.05 TB) | $0.3125 | $15.63 |
| **Savings** | **4,950 GB** | **$30.94** | **$1,546.87** |

Monthly projected savings: ~**$46,406/month**.

### 4. Materialized View Recommendation

Because this query runs 50 times per day with identical or near-identical parameters (date range fixed to "last 90 days"), a **materialized view** is the correct long-term solution:

```sql
CREATE MATERIALIZED VIEW analytics.mv_user_event_counts
PARTITION BY DATE(refresh_date)
CLUSTER BY user_id
OPTIONS (
    enable_refresh = true,
    refresh_interval_minutes = 60
)
AS
SELECT
    u.user_id,
    u.email,
    COUNT(e.event_id) AS event_count,
    CURRENT_DATE() AS refresh_date
FROM analytics.users u
LEFT JOIN analytics.events e
    ON e.user_id = u.user_id
   AND e.event_date >= DATE_SUB(CURRENT_DATE(), INTERVAL 90 DAY)
GROUP BY u.user_id, u.email;
```

The materialized view is refreshed hourly. BI dashboard reads hit the pre-aggregated result — bytes scanned drop to near zero for reads against the view. BigQuery's smart tuning also uses the materialized view automatically for compatible query rewrites, effectively acting as a persistent cache for this access pattern.

## Trade-offs

| Option | Pros | Cons |
|---|---|---|
| Partition filter + JOIN rewrite | Immediate cost reduction, no schema changes | Requires deploying query change to BI tool; risk of result set differences if subquery/JOIN semantics diverge on NULLs |
| Materialized view | Eliminates redundant compute for repeated reads; auto-rewrite for compatible queries | Adds storage cost (~negligible for aggregated result); refresh lag means data is up to 1 hour stale; requires monitoring refresh job failures |
| Cache only | Zero-effort for cached hits | Cache misses still scan 5TB; TTL-based expiry means cold-start runs remain expensive; no protection during cache invalidation events |
| Scheduled query + summary table | Full control over refresh cadence | Operational overhead; must manage table lifecycle and TTL manually |

**Chosen trade-off:** Accept up to 1 hour of staleness in exchange for ~99% cost reduction. BI dashboards displaying "last 90 days" trends are not sensitive to sub-hour freshness.

## Next Steps

1. **Immediate (today):** Rewrite the dashboard query in the BI tool to use the partitioned JOIN form. Validate row counts match the subquery version in staging before promoting.
2. **Short term (this sprint):** Create the `analytics.mv_user_event_counts` materialized view in production. Update the dashboard data source to point to the view.
3. **Monitoring:** Add a BigQuery slot usage alert (threshold: > 500 slot-hours/day on this project) and a bytes scanned alert (threshold: > 100GB per single query execution) to detect regressions.
4. **Governance:** Add a dbt schema test or query policy tag to `analytics.events` requiring that all production queries include an `event_date` partition filter. Enforce via a scheduled `INFORMATION_SCHEMA.JOBS` audit query.
5. **Index review:** Confirm clustering column order on `analytics.events` is `(event_date, user_id)` — if `user_id` is the first clustering column, reorder to `(event_date, user_id)` to align with the dominant filter pattern.
6. **Cost allocation:** Tag the materialized view refresh job with the `bi-dashboard` cost label for accurate chargeback reporting.
