# TC-E2E-021 — DATA_MODELING: Many-to-Many Bridge Table (Orders ↔ Promotions)

## Summary

A many-to-many relationship exists between `orders` and `promotions`: a single order may redeem multiple promotions simultaneously (e.g., a loyalty discount stacked with a seasonal coupon), and a single promotion may be applied across thousands of orders. Resolving this relationship in a dimensional model requires a **bridge table** (also called a factless fact table) that sits between the two dimension tables and stores the association keys at the correct grain.

This response covers the full design: grain definition, DDL for the bridge table and surrounding dimensions, cardinality analysis, BI query patterns, and guidance for dbt implementation.

---

## Decision

Model the orders-to-promotions relationship using a **bridge table** (`bridge_order_promotions`) with the grain of one row per order-promotion combination. This table is a **factless** fact table — it carries no additive numeric measures; its sole purpose is to represent membership in the many-to-many set.

```
dim_orders ──< bridge_order_promotions >── dim_promotions
                    │
                    └── (optional) dim_dates (applied_date_key)
```

**DDL — `bridge_order_promotions`**

```sql
CREATE TABLE warehouse.bridge_order_promotions (
    order_key          BIGINT       NOT NULL,  -- FK → dim_orders.order_key
    promotion_key      BIGINT       NOT NULL,  -- FK → dim_promotions.promotion_key
    applied_date_key   INT          NOT NULL,  -- FK → dim_dates.date_key (YYYYMMDD)
    discount_amount    NUMERIC(12,2),          -- optional degenerate measure
    created_at         TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT pk_bridge_order_promotions PRIMARY KEY (order_key, promotion_key)
);

CREATE INDEX idx_bridge_promo ON warehouse.bridge_order_promotions (promotion_key);
```

**DDL — `dim_promotions`**

```sql
CREATE TABLE warehouse.dim_promotions (
    promotion_key      BIGINT       NOT NULL PRIMARY KEY,
    promotion_id       VARCHAR(64)  NOT NULL,  -- natural key
    promotion_name     VARCHAR(255) NOT NULL,
    promotion_type     VARCHAR(64),            -- LOYALTY | SEASONAL | REFERRAL
    start_date         DATE,
    end_date           DATE,
    discount_pct       NUMERIC(5,2),
    is_stackable       BOOLEAN      NOT NULL DEFAULT TRUE,
    dw_created_at      TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    dw_updated_at      TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

---

## Rationale

### Grain Definition

The grain of `bridge_order_promotions` is: **one row per (order, promotion) combination, per application event**. This is the finest possible grain for this relationship and avoids the fan-out problem that arises when attempting to embed promotion data directly into `fact_orders`.

Without a bridge, two anti-patterns emerge:

1. **Comma-delimited promotion IDs in `fact_orders`** — violates 1NF, breaks SQL aggregation, and prevents dimension joins.
2. **Row duplication in `fact_orders`** — one row per promotion per order inflates all additive measures (revenue, quantity) by the number of promotions on each order, producing incorrect SUM aggregations.

### Why Factless

A **factless** fact table is the correct pattern here because:
- There is no inherent numeric measure that belongs to the association itself (the optional `discount_amount` is a degenerate measure that can be included but is not required for the table to function).
- The table answers coverage questions: "Which orders used promotion X?" and "Which promotions were applied to order Y?"
- Counting rows in the bridge is a valid measure (number of promotions per order, number of orders per promotion).

### Cardinality Analysis

| Relationship | Cardinality |
|---|---|
| `dim_orders` → `bridge_order_promotions` | One-to-many (one order, N promotion rows) |
| `dim_promotions` → `bridge_order_promotions` | One-to-many (one promotion, M order rows) |
| `dim_orders` → `dim_promotions` (via bridge) | Many-to-many (M:N) |

Expected cardinality at scale: if 30% of orders use at least one promotion and the average is 1.4 promotions per promoted order, the bridge will be approximately 0.42 times the row count of `fact_orders`. This is manageable and does not require partitioning until `fact_orders` exceeds ~500M rows.

The many-to-many relationship means you must never join `dim_orders` directly to `dim_promotions` without routing through the bridge — doing so produces a Cartesian product.

### BI Query Patterns

**How many distinct orders used each promotion?**

```sql
SELECT
    p.promotion_name,
    p.promotion_type,
    COUNT(DISTINCT b.order_key)   AS order_count,
    SUM(b.discount_amount)        AS total_discount_given
FROM warehouse.bridge_order_promotions b
JOIN warehouse.dim_promotions p
    ON b.promotion_key = p.promotion_key
JOIN warehouse.dim_dates d
    ON b.applied_date_key = d.date_key
WHERE d.calendar_month = '2026-02'
GROUP BY 1, 2
ORDER BY 3 DESC;
```

**What promotions were stacked on high-value orders?**

```sql
SELECT
    o.order_id,
    o.order_total_usd,
    COUNT(b.promotion_key)        AS promotions_used,
    ARRAY_AGG(p.promotion_name)   AS promotion_names
FROM warehouse.dim_orders o
JOIN warehouse.bridge_order_promotions b
    ON o.order_key = b.order_key
JOIN warehouse.dim_promotions p
    ON b.promotion_key = p.promotion_key
WHERE o.order_total_usd > 500
GROUP BY 1, 2
HAVING COUNT(b.promotion_key) > 1
ORDER BY 3 DESC;
```

## Trade-offs

| Approach | Pros | Cons |
|---|---|---|
| Bridge / factless fact table (chosen) | Clean grain, correct aggregations, BI-tool compatible | Requires a multi-table join in every promotion-aware query |
| Embed promotion array in `fact_orders` (JSON column) | Single-table queries | Cannot join to `dim_promotions`; breaks aggregation engines; unindexable |
| Duplicate `fact_orders` rows per promotion | No bridge join needed | All measures (revenue, quantity) are double-counted; must use DISTINCT or DIVIDE patterns everywhere |
| Separate `fact_order_promotions` with full measures | Useful if discount amounts become first-class metrics | Overengineered if promotions are purely dimensional attributes |

**BI Tool Considerations**

Most modern BI tools (Looker, Tableau, Power BI) require explicit bridge table handling to avoid fan-out from many-to-many joins:
- **Looker**: Define the bridge as a separate `view`, join with `many_to_many` relationship type, use `type: count_distinct` on order/promotion keys.
- **Power BI**: Use a bidirectional filter on the bridge relationship cautiously — enable only when querying the bridge direction that is safe; prefer unidirectional + DAX `CROSSFILTER` for ambiguous paths.
- **Tableau**: Use a logical layer relationship (not a physical join) so Tableau applies LOD calculations per domain.

## Next Steps

1. **dbt model**: Create `bridge_order_promotions` as a dbt incremental model with `unique_key = ['order_key', 'promotion_key']` to support idempotent loads. Source from the transactional `order_promotions` junction table in the operational DB.

2. **Data quality tests**: Add dbt tests — `not_null` on both keys, `relationships` test to `dim_orders` and `dim_promotions`, and a custom test asserting no `dim_orders` row has more than 10 bridge rows (business rule cap on stackable promotions).

3. **BI semantic layer**: Update the Looker / dbt Semantic Layer definitions to expose `bridge_order_promotions` with the correct many-to-many join configuration and pre-built measures (`distinct_order_count`, `distinct_promotion_count`).

4. **Backfill**: Run a one-time historical load from the transactional database for the prior 24 months to populate the bridge before enabling BI dashboards.

5. **SCD on `dim_promotions`**: Evaluate whether promotion attributes (discount percentage, type) should be SCD Type 2 to support "what was the discount at time of order" analyses. If yes, add a `promotion_snapshot_key` to the bridge pointing to the Type 2 row active on `applied_date_key`.

6. **Cardinality monitoring**: Add a dbt exposure or data observability alert if the average promotions-per-order exceeds 5, which may indicate an upstream bug in the order management system.
