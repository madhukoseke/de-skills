# TC-E2E-016 — SQL Mode: Performance Tuning a Slow Query

**Mode:** SQL
**Scenario:** Optimizing a slow query with nested joins, unbounded window functions, and full table scans lacking predicate pushdown.
**Date Captured:** 2026-03-25

---

## Summary

The query under review is a multi-step analytical report joining three large tables — `orders` (~2 billion rows), `customers` (~50 million rows), and `order_items` (~8 billion rows) — using a correlated subquery in the SELECT clause, a `ROW_NUMBER()` window function applied over the entire `orders` dataset without any partition filter, and a full scan of `order_items` with no date predicate before downstream filtering. The combined effect is catastrophic for query performance: every execution materializes billions of rows before any filtering occurs, and the correlated subquery re-executes the inner table scan for each row in the outer result set, producing near-O(n²) behavior as row counts grow.

Running `EXPLAIN (ANALYZE, BUFFERS)` against the query confirms the worst-case execution shape. The plan shows a `Seq Scan` on `orders` with an estimated cost of 9,800,000 units and actual row count of 1.94 billion. Immediately above it sits a `Nested Loop` node referencing `customers`, which re-executes the customer lookup once per order row. Above that, a `Sort` node materializes all 1.94 billion rows ordered by `created_at DESC` before the `WindowAgg` node assigns `ROW_NUMBER()` values. No `Index Scan` or `Bitmap Heap Scan` node appears anywhere in the plan, confirming zero predicate pushdown. In BigQuery terms, the Query Execution Details show 4.5 TB billed, 100% slot utilization for the duration, and a shuffle stage consuming 4.2 TB of intermediate data during the window sort.

Three distinct anti-patterns drive this profile. First, a correlated subquery of the form `(SELECT customer_name FROM customers WHERE customer_id = o.customer_id)` in the SELECT list prevents the planner from choosing a hash or merge join, forcing a nested loop re-execution per row. Second, `ROW_NUMBER() OVER (ORDER BY created_at DESC)` has no `PARTITION BY` clause, requiring the engine to globally sort every row in `orders` before assigning ranks. Third, no WHERE predicate on `order_date` is present, so the planner cannot exploit the table's monthly range partitioning and falls back to scanning all partitions.

Applying all three fixes — replacing the correlated subquery with an explicit `LEFT JOIN`, adding `PARTITION BY customer_id` to the window function, and introducing a `WHERE order_date BETWEEN @start_date AND @end_date` predicate at the innermost CTE — reduces bytes scanned from 4.5 TB to approximately 80 GB and reduces wall-clock execution time from 47 minutes to under 90 seconds in a staging environment test on January 2026 data. The EXPLAIN plan after the rewrite shows `Bitmap Index Scan` replacing `Seq Scan`, a `Hash Join` replacing `Nested Loop`, and the sort node operating on the per-customer partition subset rather than the full table. Output equivalence was verified via a row-by-row `EXCEPT` check on the 30-day sample with zero discrepancies found.

---

## Decision

Rewrite the query with three targeted changes, each addressing one identified anti-pattern:

1. **Replace the correlated subquery with an explicit JOIN.** The `(SELECT customer_name FROM customers WHERE customer_id = o.customer_id)` expression in the SELECT clause is replaced by a `LEFT JOIN customers c ON o.customer_id = c.customer_id`, with `c.customer_name` selected directly. This allows the planner to choose a hash join or merge join strategy that scales linearly.

2. **Add a `PARTITION BY` clause to the window function.** The `ROW_NUMBER() OVER (ORDER BY created_at DESC)` expression is changed to `ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY created_at DESC)`. Where a global ranking is genuinely required by the business logic, the query is first scoped to the target date window in a CTE before the window function is applied, bounding the sort domain.

3. **Add a partition-pruning predicate as early as possible.** A `WHERE order_date BETWEEN @start_date AND @end_date` clause is added in the innermost CTE or subquery referencing `orders`, enabling the storage engine to skip all monthly partitions outside the requested range. This predicate is parameterized rather than hard-coded to support reuse across scheduled runs.

This rewrite is approved for deployment. Both the correlated subquery pattern and the unbounded window function over large tables are designated as **not recommended** in the team SQL style guide and will be enforced via EXPLAIN-based CI checks going forward.

## Rationale

**Reading the EXPLAIN plan.** The most important diagnostic step before any optimization is to capture and read the full `EXPLAIN` output, including actual row counts and buffer hits. The plan tree is read bottom-up: each leaf node represents a storage access, and each interior node represents a processing step. The critical signals in this case are: `Seq Scan` (full table access, no index), `Nested Loop` with re-scanned inner side (correlated join), `Sort` over billions of rows (unbounded window), and the absence of any `Partition Pruning` annotation. Once these are identified, the fix strategy becomes clear.

In BigQuery, the equivalent is the Query Execution Details panel, which shows per-stage slot time, bytes shuffled, and the most expensive stage by wall-clock time. Here, Stage 2 (the shuffle for the window sort) accounts for 38 of the 47 total minutes. Stage 1 (the nested loop equivalent, implemented as a broadcast join fallback for the correlated subquery) accounts for another 7 minutes. Fixing the two most expensive stages first maximizes return on effort.

**Correlated subquery as a join anti-pattern.** A correlated subquery in the SELECT clause is syntactic sugar for "for each row in the outer query, execute this inner query." The SQL standard allows this, and most planners can in principle detect and rewrite it as a join — but this rewrite is only reliable when the inner query is provably non-correlated (i.e., does not reference outer scope columns in a way that changes the result set). In practice, production planners are conservative and frequently leave the correlated execution path in place, especially for complex expressions. The safe fix is to make the join explicit, which gives the planner unambiguous permission to use any join algorithm and eliminates any risk of the nested-loop anti-pattern appearing.

**Window function partition scoping.** A window function with no `PARTITION BY` clause treats the entire input as one partition. For `ROW_NUMBER()`, this means every row in `orders` must be assigned a position in a single global ranking, which requires a complete sort of the dataset. Adding `PARTITION BY customer_id` delegates the sort to per-customer subsets. With an average of 40 orders per customer, the sort complexity drops from O(N log N) over 2 billion rows to O(K × M log M) where K is the number of distinct customers (~50 million) and M is the average orders per customer (~40). The practical result is that each sort unit is tiny and the operations can be parallelized across many executors without a global shuffle.

**Partition pruning via predicate pushdown.** Range-partitioned tables (by `order_date` monthly) store each partition's data separately at the storage layer. When the query planner sees a filter on the partition key column, it can eliminate all partitions outside the filter range before reading any data — this is partition pruning. Without a predicate, the planner cannot prune anything and reads all partitions. With `WHERE order_date >= '2026-01-01' AND order_date < '2026-04-01'`, the planner reads only the 3 relevant monthly partitions out of ~84 total (7 years of history). The EXPLAIN plan change is visible as the `Seq Scan` being replaced by a `Bitmap Heap Scan` with an `Index Cond` on `order_date`. In BigQuery, this appears as a reduction from "Table: FULL" to "Table: PARTITIONED (3 partitions)" in the query execution metadata.

**Correctness validation after the fix.** Any query rewrite carries the risk of changing results, not just performance. The validation methodology used here is: (1) run both the original and rewritten queries on the same 30-day sample date range, (2) compute `original_results EXCEPT rewritten_results` and `rewritten_results EXCEPT original_results`, and (3) assert both EXCEPT results return zero rows. This confirms the rewrite is semantically equivalent for the tested range. Additionally, the rewrite changes the window function scope (from global to per-customer), which is a semantic change the team confirmed was the intended behavior all along — the original query was producing incorrect global rankings due to the missing `PARTITION BY`.

## Trade-offs

| Approach | Bytes Scanned | Wall-Clock Time | SQL Complexity | Regression Risk |
|---|---|---|---|---|
| Original (correlated subquery + unbounded window + no predicate) | ~4.5 TB | ~47 min | Low (easy to read) | Baseline |
| Predicate pruning only | ~350 GB | ~8 min | Low | Low |
| Explicit JOIN only (no window or predicate fix) | ~4.5 TB | ~22 min | Low-Medium | Low |
| PARTITION BY window only (no join or predicate fix) | ~4.5 TB | ~18 min | Low | Low |
| All three fixes combined (recommended) | ~80 GB | ~90 sec | Medium | Low |
| Materialize intermediate CTE as temp table | ~80 GB | ~2 min | High | Low-Medium |

**Correlated subquery vs. explicit JOIN.** The correlated subquery reads naturally and is easy for analysts unfamiliar with JOIN syntax to understand. However, it is not recommended in production workloads on large tables because it prevents the planner from choosing efficient join algorithms. An explicit JOIN is marginally more verbose but unambiguously correct in its semantics and gives the planner full algorithmic freedom. The only acceptable use of a correlated subquery in production is when the inner query is provably small (e.g., a scalar subquery against a single-row config table), and even then an explicit JOIN is preferred for consistency.

**Partitioned window vs. global window.** Adding `PARTITION BY` to the window function changes its semantics: `ROW_NUMBER()` now ranks within each customer's orders rather than globally. If global ranking is a genuine business requirement, the workaround is to pre-filter the dataset to a bounded date range in a CTE before applying the global window. This preserves global ranking semantics while bounding the sort domain to a manageable subset. The trade-off is that results outside the date range are excluded, which is acceptable for most reporting use cases but must be explicitly confirmed with the stakeholder.

**Dynamic vs. static date predicates.** Parameterized date expressions (`@start_date`, `@end_date`) are preferred for scheduled queries because they avoid manual updates when the reporting window rolls forward. Dynamic expressions (`DATE_TRUNC('month', CURRENT_DATE) - INTERVAL '3 months'`) are even lower maintenance but can make debugging harder — when a query returns unexpected results, it is not immediately obvious what date range was applied. The recommended pattern is to use parameters for scheduled jobs and to log the resolved parameter values in the job execution metadata.

**Incremental temp table materialization.** For very complex multi-CTE queries, materializing an intermediate CTE result into a temp table (and building an index on it) can outperform in-query CTE execution if the CTE is referenced multiple times and the planner does not automatically cache it. This approach is not recommended here because the rewritten query references each CTE only once, making materialization unnecessary overhead. It would be the right choice if the query structure required re-joining the same filtered `orders` subset multiple times.

## Next Steps

1. **Deploy rewritten query to production** — Replace the view definition of `reporting.slow_orders_report` with the rewritten SQL in the next maintenance window (target: within 5 business days). Assign to: data engineering team lead. Requires: peer review of the rewritten SQL against the validation test results.

2. **Add EXPLAIN-based CI gate** — Integrate a lightweight plan-inspection script into the SQL CI pipeline that runs `EXPLAIN` on all queries under `reporting/` and fails the build if any query shows a `Seq Scan` on tables with more than 100 million rows. This prevents the partition-pruning regression from being reintroduced silently.

3. **Update SQL style guide** — Add a dedicated "Anti-patterns" section documenting: (a) correlated subqueries in SELECT clauses are not recommended for large tables; (b) window functions must include a `PARTITION BY` clause or be explicitly scoped to a pre-filtered CTE; (c) every query touching a partitioned table must include a predicate on the partition key. Link TC-E2E-016 as the reference example.

4. **Audit all reporting schema views** — Run the EXPLAIN-based scan across all 12 views in the `reporting` schema to identify other instances of the same anti-patterns. Produce a prioritized list ordered by estimated bytes-scanned impact. Target completion: 2 weeks.

5. **Evaluate clustering key addition** — Assess whether adding `CLUSTER BY customer_id` (BigQuery) or a composite index on `(order_date, customer_id)` (PostgreSQL) would further reduce scan cost for the fixed query. This is a secondary optimization — the three primary fixes deliver the most impact, but clustering can provide an additional 20–40% reduction in bytes read for queries that filter on both columns.

6. **Configure query cost alerting** — Set up a BigQuery slot-usage or bytes-billed alert that triggers when any single query exceeds 200 GB scanned. Route the alert to the data engineering Slack channel with the query ID and execution metadata. This provides an early warning system for future cost regressions before they affect production SLAs or billing thresholds.
