# TC-E2E-029: PR_REVIEW ‚Äî Production Readiness Check for New Daily Warehouse Load Task

**Mode:** PR_REVIEW
**Date:** 2026-03-25
**PR:** `feat/add-daily-warehouse-load` ‚Äî adds a new daily BigQuery LOAD job to the ingestion DAG
**Reviewer:** Data Engineering Best Practices Skill v2

---

## Summary

This PR introduces a new Airflow task (`load_daily_events_to_bigquery`) that runs a BigQuery LOAD job from GCS to the `analytics.events` partitioned table. The implementation correctly handles partition configuration and uses a standard GCS-to-BQ operator. However, the PR has two hard FAIL items (missing retry configuration, no row-count assertion), one critical security FAIL item (hardcoded service account key), and one WARN item (no cost estimate for the new query pattern). A `REQUEST_CHANGES` verdict is issued. The PR must not be merged until all FAIL items are resolved.

---

## Decision

**REQUEST_CHANGES** ‚Äî Three FAIL items must be remediated before this PR can be approved. The hardcoded service account key is a security violation that must be addressed immediately; if this branch has been pushed to a shared remote, the key must be considered compromised and rotated now.

---

## Rationale

### Review Table

| # | Item | Status | Severity | Notes |
|---|---|---|---|---|
| 1 | Partition handling ‚Äî `write_disposition=WRITE_APPEND`, `time_partitioning` set correctly on destination table | **PASS** | ‚Äî | Partition config matches the `analytics.events` schema. `$` partition decorator used correctly. |
| 2 | Retry configuration ‚Äî `retries` and `retry_delay` set on the operator | **FAIL** | HIGH | No `retries` parameter set. Default is `0` in the custom base operator used here. A transient GCS/BQ API error will cause an immediate DAG failure with no self-recovery. |
| 3 | Row-count assertion ‚Äî downstream task validates loaded row count | **FAIL** | HIGH | No post-load assertion task exists. Silent empty loads (0 rows loaded due to upstream issues) will not be detected until a downstream consumer reports missing data, potentially hours later. |
| 4 | Hardcoded service account key in DAG file | **FAIL** | CRITICAL ‚Äî security risk | `sa_key = '{"type": "service_account", ...}'` is hardcoded in `dags/daily_load.py` line 47. This is a plain-text credential in a version-controlled file. Immediate rotation required. |
| 5 | Cost estimate for new query/load pattern | **WARN** | MEDIUM ‚Äî cost impact | No cost estimate provided in the PR description. The LOAD job itself is free in BigQuery, but the downstream transformation query triggered by the load is estimated at ~500GB/run. At $6.25/TB, this is ~$3.13/day / ~$94/month. Should be acknowledged. |
| 6 | DAG schedule alignment with upstream data availability | **PASS** | ‚Äî | `schedule_interval="0 6 * * *"` (06:00 UTC) is correctly offset 2 hours after the upstream GCS export completes at 04:00 UTC. |
| 7 | Idempotency ‚Äî is the load operation safe to re-run? | **PASS** | ‚Äî | `write_disposition=WRITE_TRUNCATE` on the partition (not the full table) ensures re-running the task overwrites only the target date partition. Idempotent. |
| 8 | Connection reference ‚Äî uses Airflow connection ID, not hardcoded project | **FAIL** | CRITICAL | See item 4 ‚Äî the service account key replaces what should be a `gcp_conn_id` reference to an Airflow connection. This is a duplicate dimension of the security failure. |

### Risk Assessment

**Overall risk score: HIGH (7/10)**

- **Security risk (10/10):** Hardcoded service account key. If this commit is on any remote branch (GitHub, GitLab, etc.), the key must be treated as compromised immediately.
- **Reliability risk (7/10):** No retries + no row-count assertion means both transient failures and silent data loss go undetected.
- **Cost risk (3/10):** Manageable (~$94/month) but should be documented for budget tracking.

### Required Fixes (Blocking)

#### Fix 1: Remove hardcoded credential ‚Äî security

```python
# BEFORE (current ‚Äî DO NOT MERGE)
sa_key = '{"type": "service_account", "project_id": "my-project", ...}'

# AFTER (required fix)
load_task = GCSToBigQueryOperator(
    task_id="load_daily_events_to_bigquery",
    gcp_conn_id="gcp_prod_conn",          # Airflow connection ID ‚Äî key stored in Secrets Backend
    ...
)
```

The `gcp_prod_conn` Airflow connection must be configured in the Secrets Backend (GCP Secret Manager or HashiCorp Vault), not in the Airflow UI metadata database. Rotate the exposed key before deploying the fix.

#### Fix 2: Add retry configuration

```python
load_task = GCSToBigQueryOperator(
    task_id="load_daily_events_to_bigquery",
    retries=3,
    retry_delay=timedelta(minutes=5),
    retry_exponential_backoff=True,
    max_retry_delay=timedelta(minutes=30),
    ...
)
```

#### Fix 3: Add row-count assertion task

```python
from airflow.providers.google.cloud.operators.bigquery import BigQueryCheckOperator

assert_row_count = BigQueryCheckOperator(
    task_id="assert_daily_events_row_count",
    sql="""
        SELECT COUNT(*) > 0
        FROM `my-project.analytics.events`
        WHERE DATE(_PARTITIONTIME) = DATE('{{ ds }}')
    """,
    use_legacy_sql=False,
    gcp_conn_id="gcp_prod_conn",
)

# Task order: load ‚Üí assert ‚Üí downstream
load_task >> assert_row_count >> downstream_transform_task
```

### Recommended Fix (Non-blocking WARN)

#### Fix 4: Add cost estimate to PR description

Add the following to the PR description:

> **Cost impact:** The LOAD job is free (BigQuery charges for storage only). The downstream `transform_daily_events` query scans ~500GB/run. At $6.25/TB on-demand pricing: $3.13/run √ó 1 run/day = ~$94/month. Tagged to cost label `de-ingestion` for chargeback tracking.

## Trade-offs

| Approach | Pros | Cons |
|---|---|---|
| Block merge until all FAILs resolved | Prevents security incident and reliability issues from reaching production | Delays the feature; engineer must do rework |
| approve with comments on FAILs | Faster shipping | Unacceptable given the hardcoded credential ‚Äî this is a mandatory block |
| Approve with WARN on cost only | Reasonable for non-blocking cost documentation | Only appropriate after security and reliability FAILs are resolved |
| Auto-rotate key via CI secret scanner | Catches this class of issue earlier (pre-commit) | Requires pre-commit hook setup (`detect-secrets`, `truffleHog`) ‚Äî not in place yet; should be added as a follow-up |

**Decision:** No trade-off is acceptable for the security FAIL. The other FAILs are also hard blocks given the reliability standards documented in the DE playbook (all production tasks must have retries and output assertions).
