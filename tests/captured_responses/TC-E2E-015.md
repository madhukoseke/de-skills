## Summary

The e-commerce orders domain maps naturally to a Kimball star schema with one central fact table (`fact_orders`) and three conformed dimensions (`dim_customers`, `dim_products`, `dim_date`). Given that queries filter by `order_date` and aggregate by `customer_tier` and `product_category`, this is the correct choice over OBT (which would make independent dimension updates expensive) or Data Vault (unnecessary complexity for this team size and use case).

## Decision

Build a **Kimball star schema** with:
- `fact_orders` — grain: one row per order line item
- `dim_customers` — SCD Type 2 for `customer_tier` (changes affect historical analysis)
- `dim_products` — SCD Type 1 for `product_category` (overwrite; category history not needed)
- `dim_date` — static, pre-populated for 10 years

## Rationale

**Why Kimball over OBT:** `dim_customers` and `dim_products` change independently of orders. If `customer_tier` changes for 50K customers, an OBT would require rewriting 50K order rows. In a star schema only the dimension row is updated.

**Why SCD Type 2 on `dim_customers.customer_tier`:** Revenue-by-tier analysis must reflect what tier the customer *was* at order time, not today's tier. SCD Type 2 preserves that history via `valid_from / valid_to / is_current`.

**Why SCD Type 1 on `dim_products.category`:** Product category is used only for present-day reporting slices; stakeholders don't ask "what was this product's category in 2022?". SCD Type 1 (overwrite) keeps the model simple.

**Why pre-build `dim_date`:** Every `fact_orders` query involves date filtering or grouping. Pre-populating `dim_date` eliminates ad-hoc `DATE_TRUNC` and `EXTRACT` expressions in every downstream query and enables consistent fiscal calendar logic in one place.

## DDL

```sql
-- fact_orders: grain = one order line item
CREATE TABLE mart.fact_orders (
    -- Surrogate keys
    order_line_sk   BIGINT      NOT NULL,
    customer_sk     BIGINT      NOT NULL REFERENCES mart.dim_customers(customer_sk),
    product_sk      BIGINT      NOT NULL REFERENCES mart.dim_products(product_sk),
    date_sk         INTEGER     NOT NULL REFERENCES mart.dim_date(date_sk),

    -- Natural/business keys (for debugging and idempotent loads)
    order_id        VARCHAR(50) NOT NULL,
    order_line_id   VARCHAR(50) NOT NULL,

    -- Measures (fully additive)
    quantity        INTEGER     NOT NULL,
    unit_price      NUMERIC(12, 4) NOT NULL,
    discount_amount NUMERIC(12, 4) NOT NULL DEFAULT 0,
    total_amount    NUMERIC(12, 4) NOT NULL,

    -- Degenerate dimensions
    order_channel   VARCHAR(50),
    payment_method  VARCHAR(50),

    -- Audit
    _loaded_at      TIMESTAMP   NOT NULL DEFAULT CURRENT_TIMESTAMP,
    _source         VARCHAR(200)
);

-- dim_customers: SCD Type 2 on customer_tier
CREATE TABLE mart.dim_customers (
    customer_sk     BIGINT      PRIMARY KEY GENERATED ALWAYS AS IDENTITY,
    customer_id     VARCHAR(50) NOT NULL,
    name            VARCHAR(200),
    email           VARCHAR(200),
    country_code    CHAR(2),
    customer_tier   VARCHAR(20),          -- tracked historically (SCD Type 2)
    -- SCD Type 2 fields
    valid_from      TIMESTAMP   NOT NULL,
    valid_to        TIMESTAMP,            -- NULL = current record
    is_current      BOOLEAN     NOT NULL DEFAULT TRUE,
    _loaded_at      TIMESTAMP   NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- dim_products: SCD Type 1 (overwrite)
CREATE TABLE mart.dim_products (
    product_sk      BIGINT      PRIMARY KEY GENERATED ALWAYS AS IDENTITY,
    product_id      VARCHAR(50) NOT NULL UNIQUE,
    product_name    VARCHAR(200),
    category        VARCHAR(100),         -- overwritten on change (SCD Type 1)
    subcategory     VARCHAR(100),
    unit_cost       NUMERIC(12, 4),
    is_active       BOOLEAN     NOT NULL DEFAULT TRUE,
    _loaded_at      TIMESTAMP   NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- dim_date: static, pre-populated
CREATE TABLE mart.dim_date (
    date_sk         INTEGER     PRIMARY KEY,  -- YYYYMMDD (e.g., 20240115)
    full_date       DATE        NOT NULL,
    year            SMALLINT    NOT NULL,
    quarter         SMALLINT    NOT NULL,
    month           SMALLINT    NOT NULL,
    month_name      VARCHAR(20) NOT NULL,
    week_of_year    SMALLINT    NOT NULL,
    day_of_week     SMALLINT    NOT NULL,     -- 1=Monday
    day_name        VARCHAR(20) NOT NULL,
    is_weekend      BOOLEAN     NOT NULL,
    is_holiday      BOOLEAN     NOT NULL DEFAULT FALSE,
    fiscal_year     SMALLINT,
    fiscal_quarter  SMALLINT
);
```

## Trade-offs

| Paradigm | Pros | Cons | When to Use |
|----------|------|------|-------------|
| Kimball star schema (**recommended**) | Fast aggregation queries, BI-friendly, clear grain | More tables than OBT | Known query patterns, BI-facing analytics |
| OBT (One Big Table) | Simplest queries, no joins | Customer/product updates force full order re-load | Single entity, rarely changing attributes |
| Data Vault | Full audit trail, handles schema volatility | High complexity, large team needed | Regulated industries, 5+ source systems |
