## Summary

The submitted PySpark job contains four high-severity issues: `df.collect()` on a large dataset risks OOM on the driver, a Python UDF for a string transform bypasses Catalyst optimization, the streaming query has no checkpoint, and `repartition(1)` before write serializes output to a single thread. All issues are fixable without changing the job's business logic. Adaptive Query Execution (AQE) is also not enabled.

## Decision

Replace `collect()` with a `write` or aggregation. Replace the Python UDF with the equivalent `pyspark.sql.functions` built-in. Add `checkpointLocation` to the streaming write. Change `repartition(1)` to `coalesce(N)` sized to the output volume. Enable AQE globally.

## Rationale

**`df.collect()` on large dataset:** `collect()` pulls all rows to the driver node's JVM heap. For datasets > 1 GB this reliably causes OOM. The fix depends on intent:

```python
# If writing: use df.write directly — no driver memory involved
df.write.mode("overwrite").parquet("s3://bucket/output/")

# If aggregating: keep it in Spark
summary = df.groupBy("status").agg(F.count("*").alias("cnt"))
summary.write.mode("overwrite").parquet("s3://bucket/summary/")

# If you genuinely need a small sample on the driver:
sample = df.limit(1000).collect()  # explicit and bounded
```

**Python UDF:** Python UDFs serialize each row to Python, apply the function, and deserialize back. For simple string transforms this adds 5–10× latency versus Catalyst-optimized built-ins:

```python
# Python UDF (anti-pattern):
from pyspark.sql.functions import udf
upper_udf = udf(lambda s: s.upper() if s else None, StringType())
df = df.withColumn("name_upper", upper_udf(F.col("name")))

# Built-in (recommended):
import pyspark.sql.functions as F
df = df.withColumn("name_upper", F.upper(F.col("name")))
```

**Missing checkpoint:** Without `checkpointLocation` a streaming query cannot recover from a restart. All progress is lost and the job re-reads from the earliest offset, causing data duplication or loss depending on the sink:

```python
query = (
    parsed.writeStream
    .format("delta")
    .outputMode("append")
    .option("checkpointLocation", "s3://bucket/checkpoints/orders_stream/")  # required
    .trigger(processingTime="60 seconds")
    .toTable("raw.orders_placed")
)
```

**`repartition(1)` before write:** This forces a full shuffle to a single partition and writes a single output file — serializing what could be parallel writes across executors. Use `coalesce` to reduce file count without a shuffle, or omit the repartition and let Spark write one file per partition:

```python
# Problematic:
df.repartition(1).write.parquet(output_path)

# Better (no shuffle, reduces file count):
df.coalesce(10).write.parquet(output_path)
```

**AQE not enabled:**

```python
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")
```

## Trade-offs

| Option | Pros | Cons | When to Use |
|--------|------|------|-------------|
| `df.write` instead of `collect` | No driver OOM, full parallelism | Requires output path | Any large dataset write |
| Built-in functions over UDF | Catalyst-optimized, vectorized | Requires knowing the equivalent built-in | All standard transforms |
| `coalesce` vs `repartition(1)` | No shuffle overhead | Can produce uneven file sizes | Reducing output file count |
