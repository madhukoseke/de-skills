## Summary
Schema updates should be backward compatible and versioned.

## Decision
Introduce additive optional fields and publish a new schema version in the contract.

## Rationale
Consumer stability improves when additive changes avoid breaking existing integrations.

## Trade-offs
| Option | Pros | Cons | When to Use |
|--------|------|------|-------------|
| Additive optional fields | Safe rollout | Null/default handling | Active consumer ecosystems |
| Forced field change | Strong enforcement | Break risk | Coordinated upgrades |

2. Set optional defaults.
3. Notify consumers.
