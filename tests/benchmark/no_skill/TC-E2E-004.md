## Summary
Cluster-only design for a very large table is an anti-pattern.

## Decision
Reject cluster-only approach and add partitioning first, then cluster within partitions.

## Rationale
Partitioning provides major scan pruning benefits. Clustering alone cannot control large scan costs.

## Trade-offs
| Option | Pros | Cons | When to Use |
|--------|------|------|-------------|
| Partition + cluster | Strong scalability | Requires workload tuning | Large tables |
| Cluster only | Some ordering benefit | Not recommended at this scale | Rare edge cases |
