## Summary
The SQL should be revised for idempotency and reviewed for performance before production use.

## Decision
Use an idempotent merge or delete-and-insert pattern instead of append-only writes, and review the query plan with explain to identify expensive operations.

## Rationale
Idempotent writes prevent duplicate outcomes on retries, and explain output helps detect anti-pattern query shapes such as inefficient joins or correlated scans.

## Trade-offs
| Option | Pros | Cons | When to Use |
|--------|------|------|-------------|
| MERGE-based load | Strong idempotency | More complex SQL | Incremental upserts |
| DELETE + INSERT | Simpler control flow | More table churn | Partition refresh flows |
